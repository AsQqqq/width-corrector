1Soft-body basics

BeamNG runs on a soft-body physics engine. A vehicle is not a rigid object: it is a live structure of nodes and beams that flexes, deforms and breaks under real forces.

  • Node - a point mass. It carries weight and handles most collisions. Nodes can move freely in space.
  • Beam - a spring-like connector between exactly two nodes. Beams have no mass; they define stiffness (beamSpring), damping (beamDamp), how far they bend before permanent damage (beamDeform) and when they snap (beamStrength).

A full car is split into many parts (chassis, suspension, engine, body panels, wheels...), each described in its own JBeam file. Parts connect through a slot system, which is what makes a vehicle configurable and tunable.

Key intuition: a plain box of beams collapses, because beams can pivot at the nodes. Rigidity comes from diagonal cross-bracing - triangulate your structure.

2Coordinate system

BeamNG uses a Z-up system in SI units (meters, kilograms). The vehicle axes are not the ISO/robotics convention, so this trips up almost every beginner:

AxisDirection
+XLeft (so right is -X)
+YBackward (so forward is -Y)
+ZUp

Node coordinates in JBeam are meters relative to the vehicle. Increasing X moves a node left, increasing Y moves it toward the rear, increasing Z moves it up. The vehicle frame is anchored at the refNodes (usually near the body center, low down).

World space (fixed to the map) and vehicle space (relative to the car) are different frames; gameplay/sensor systems accept either, so keep track of which one you are in.

3Folder layout & files

Unpacked mods live in the game's mods/unpacked/ folder. Prefix the mod name with your username to keep it unique. Vehicle files sit under vehicles/<name>/:

%localappdata%\BeamNG.drive\<version>\mods\unpacked\
└─ username_mymod/
   └─ vehicles/
      └─ username_mymod/
         ├─ mymod.jbeam          # root part (slotType "main")
         ├─ mymod_engine.jbeam   # more parts...
         ├─ mymod.dae            # 3D model (mesh)
         ├─ main.materials.json  # materials / textures
         ├─ info.json            # vehicle entry for the selector
         └─ mymod.pc             # a saved configuration
FileWhat it is
*.jbeamStructure & physics of a part: nodes, beams, slots, mesh binding. The core of a mod.
*.pcA saved configuration: which part fills each slot, plus paint and variables. Created in-game via Save&Load.
info.jsonVehicle entry for the car selector: brand, name, default config, paints.
info_<config>.jsonPer-configuration data: display name and specs (power, weight, drivetrain) used for filtering.
*.daeThe 3D model (Collada), usually authored in Blender. Bound to nodes via flexbodies.
*.materials.jsonMaterial definitions that reference the texture files.
*.dds / *.pngTextures used by the materials.

4JBeam syntax

JBeam is a relaxed JSON format. It is JSON, but friendlier: // and /* */ comments are allowed, and commas between elements are optional. It is case sensitive. A file is a flat dictionary whose keys are part names:

{
  "part_name": {
    "slotType": "main",   // the root part is spawned directly
    "information": { "name": "My Part", "authors": "You" },
    "nodes": [ ... ],
    "beams": [ ... ]
  }
}

Tables (spreadsheet style)

Most sections are tables: the first row names the columns, the following rows are data.

"nodes": [
  ["id", "posX", "posY", "posZ"],   // header row
  ["n1",  0.5, -1.2, 0.3],
  ["n2",  0.7, -1.5, 0.2]
]

A column name ending with : is a section link - it references ids from another section (e.g. beams reference node ids):

"beams": [
  ["id1:", "id2:"],   // the ":" means "these are node ids"
  ["n1",   "n2"]
]

Modifiers (property objects)

An object {...} placed among rows sets properties for the rows after it, until you reset it. This is how you avoid repeating values on every line:

"nodes": [
  ["id", "posX", "posY", "posZ"],
  {"nodeWeight": 25, "group": "body"},   // applies below
  ["n1", 0.0, -1.5, 0.0],
  ["n2", 0.37, -0.98, 0.22, {"nodeWeight": 3}],  // row-only override
  {"group": ""}                             // reset the group
]

Dictionaries (plain key/value objects) are used for one-off data such as camera or engine settings, instead of the table form.

5Parts & the slot system

Every part declares a slotType. A parent part defines slots that accept children by their slotType, so any matching part can fill a slot. Exactly one part uses slotType = "main": it is the root, loaded first.

main (root part)
├─ slot "mymod_engine"    → any part with that slotType
├─ slot "mymod_suspension_F" → front suspension options
└─ slot "mymod_wheels_F"     → front wheel options

The slots section lists, for each slot, its slotType, the default part to load, and a description shown in the UI. This tree is exactly what the parts selector and the tuning menu show, and what a .pc file records.

6Section reference

The sections you will meet most often inside a part:

SectionPurpose
informationDisplay name, authors, value/price of the part.
slotType / slotsWhere this part fits, and which child parts it accepts.
nodesPoint masses: positions, weight, collision and grouping.
beamsConnections between nodes: spring, damp, deform, strength.
trianglesSurfaces between three nodes for aerodynamics and collision.
flexbodiesBind the visible mesh to nodes so the model deforms with the structure.
hydrosBeams whose length changes on command (steering, actuators).
rails / slidenodesNodes that slide along a rail (steering racks, rigidity).
torsionbarsRotational resistance between lever arms (anti-roll bars).
refNodesReference nodes that set vehicle orientation and camera anchoring.
camerasInterior/exterior camera positions.
propsNon-deforming animated meshes (gauges, pedals, needles).
powertrain / engineDrivetrain: engine torque curve, gearbox, differentials.
variablesTunable values exposed in the UI (e.g. spring rates).
controller / electricsLua controllers and the electrics value bus (lights, signals).

7Models & materials

The visible car is a 3D mesh (usually a .dae from Blender). It is not physical by itself - it is attached to the node structure through the flexbodies section, so it bends and breaks with the beams underneath.

  • Origin matters. Export the mesh with the origin aligned to the vehicle, or the model will appear offset from its nodes.
  • Enough nodes. A flexbody needs at least 3 non-collinear nodes that surround the mesh, or you get "VY node" errors and bad deformation.
  • Materials. *.materials.json maps material names used in the mesh to texture files (.dds/.png) and shader settings.

8Configurations & info files

A .pc file is a saved vehicle setup: the part chosen for each slot, paint colors and variable values. You create one in-game with Save&Load and drop it in the mod folder to ship a preset.

  • info.json - the vehicle's main entry: brand, name, available paints, and default_pc (which config loads by default).
  • info_<config>.json - one per configuration: its display name plus stats (power, torque, weight, drivetrain, 0-100, top speed) used for filtering and the info panel.

9Common issues

  • Missing quotes around strings (especially node ids) break parsing - use a syntax-highlighting editor.
  • Unmatched brackets. The game console points at the file and rough location of the mismatch.
  • Duplicate part names - the second silently overwrites the first. Keep names unique.
  • Flexbody offset - a wrong Blender origin puts the mesh in the wrong place.
  • Missing mesh reference - a typo in the flexbody name or a missing .dae stops the model loading.
  • Too few flexbody nodes - needs 3+ non-collinear nodes surrounding the mesh.
  • No bracing - a part with no diagonal beams collapses on spawn.
  • Beam values out of balance - huge spring/damp vs node weight causes vibration and instability.
  • Nodes clipping on spawn push panels outward; reposition nodes or tune self-collision.
  • Console errors (duplicate beams, orphan references) hurt performance - audit the console before release.

10Tips & best practices

  • Indent with spaces and align ids/coordinates in columns - it makes vertical/batch edits safe and fast.
  • Limit find/replace to the selected lines so you do not accidentally change JBeam modifiers elsewhere.
  • Debug with beam visualization set to Deformation to find over-stressed beams; disable gravity and drivetrain to isolate structural problems.
  • Watch collision cost - keep ground-collision structures sparse, use NONCOLLIDABLE triangles in dense node clusters, and keep at least ~3 cm between nodes and triangles.
  • Break rails and torsion bars through small dummy beams placed between their nodes.
  • Start small. A simple part with few moving pieces teaches the workflow fastest.

11Tools & workflow

  • Blender - free 3D modeling for the mesh and for placing nodes.
  • VS Code + the official JBeam extension - editing with validation and highlighting.
  • An image editor (GIMP / Paint.NET) for textures.
  • The in-game console and debug views - your main feedback loop for errors and structural stress.

WidthCorrector itself helps here: batch-tune node weight and beam parameters across a whole folder, inspect the 3D node/beam frame, edit engine curves, and edit raw files in the built-in code editor.
