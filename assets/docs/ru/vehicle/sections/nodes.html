Ноды — это точечные массы и основа физики BeamNG. Всё построено вокруг них. Нода сама по себе — это не имеющая размеров (бесконечно малая) точечная масса в трёхмерном пространстве. Каждая нода определяется уникальным именем и позицией в 3D-пространстве.

Имя ноды может быть любой строкой, лишь бы имена не конфликтовали. Хорошая схема именования: запоминающаяся/подходящая буква, затем цифра, затем буква, обозначающая сторону автомобиля. Например, первую крайнюю правую ноду рамы можно назвать f1rr, а третью левую ноду рамы — f3l. Нода, находящаяся посередине (posX = 0), не получает буквенного суффикса.

Ниже приведён пример секции нод. У нод есть множество свойств, которые задаются строками-словарями, расположенными выше тех нод, на которые они должны действовать. Вес нод указывается в килограммах, координаты — в метрах. Модификатор «group» предназначен для назначения флексбоди. Материал ноды влияет только на звуковые и частичные эффекты, но не на физику.

Наш игровой движок использует систему координат Z-вверх и единицы СИ (метры, килограммы) для большинства параметров.

Важно — координаты нод используют пространство транспортного средства BeamNG, а не +X вперёд.

Каждое значение posX, posY и posZ в JBeam задаётся относительно системы отсчёта refnode транспортного средства: +X = влево, +Y = назад, +Z = вверх; вперёд = −Y. Не считайте, что +X — это направление движения, а +Y — это боковое направление; такое предположение приведёт к неверному расположению нод. Подробнее см. в разделе Системы координат с полным описанием системы отсчёта и схемами.


Система коллизий

В BeamNG коллизия обрабатывается исключительно нодами. (Балки не могут ни с чем сталкиваться.)

Существует 3 типа коллизии нод:

  • Heightmap (с землёй)
  • Static (с полигонами коллизии объектов карты)
  • Dynamic (с треугольниками транспортного средства), который дополнительно делится на:
    • Self-collision (треугольники того же транспортного средства, которому принадлежит нода)
    • External collision (треугольники других транспортных средств)

Коллизия с картой высот отличается от статической наличием нескольких слоёв, что помогает шинам транспортного средства вращаться плавнее. Типы коллизии имеют приоритет: динамическая коллизия имеет приоритет над статической и коллизией с картой высот. На практике это означает, что при достаточно большой силе нода может слегка проникнуть сквозь твёрдую стену или землю, а затем выйдет обратно, как только (поверхностный) треугольник перестанет её толкать. Коллизия с картой высот также оказывает меньшее влияние на производительность, чем статическая — поэтому для транспортного средства, которое будет часто ездить по ассетам карты (например, по камням), а не по обычной дорожной разметке, важнее иметь меньше нод снизу.

В Jbeam эти типы управляются тремя свойствами:

  • Аргумент collision влияет на все типы коллизий и имеет наивысший приоритет — включение любого другого типа коллизии не сработает, если этот параметр отключён.
  • Аргумент selfCollision влияет только на тип самостолкновения в рамках динамической коллизии.
  • Аргумент staticCollision влияет на статическую коллизию и коллизию с картой высот.

Со стороны (поверхностных) треугольников , когда нода приближается к треугольнику, на неё начинают действовать силы коллизии, которые пытаются остановить ноду, и сила anti-clip, определяющая, с какой стороны треугольника окажется нода. Если нода приближается к треугольнику спереди (отображается зелёным при включённой визуализации треугольников), силы начинают действовать примерно на расстоянии 2,5 см от треугольника, чтобы предотвратить проникновение. Управлять этими силами можно с помощью соответствующих свойств. Свойство externalCollisionBias смещает расстояние, на котором начинают действовать силы. Когда при загрузке сцены много нод сталкиваются с треугольниками, это может заметно снизить производительность, поэтому лучше поддерживать расстояние около 3 или 3,5 см между ними.

Когда triangleType установлен в NONCOLLIDABLE, треугольник не будет участвовать в коллизии, однако система anti-clip для него сохраняется. Все ноды с включённой коллизией должны иметь между собой хотя бы NONCOLLIDABLE-треугольники, чтобы автомобили не застревали друг в друге при столкновении. Треугольник типа NORMALTYPE не оказывает дополнительного влияния на производительность, если в радиусе 50 см от его поверхности нет нод, однако каждая такая нода добавляет дополнительную нагрузку. Треугольник типа NONCOLLIDABLE не имеет таких последствий для производительности, поэтому предпочтительнее использовать их в небольших конструкциях с плотно расположенными нодами, например во внедорожных фарах.

В целях оптимизации и проектирования: группы нод (groups) не позволяют ноде сталкиваться с треугольниками, образованными нодами той же группы (аргумент group), даже если для selfCollision установлено значение true. Если хотя бы одна из нод принадлежит нескольким группам (groups), для каждой ноды учитывается первая группа из списка. Например, нода с группами {"A", "B"} будет сталкиваться с треугольниками, образованными нодами с группами {"C", "B"}, но не с треугольниками, образованными нодами с группами {"A", "C"}. Эта система может работать не так, как ожидается, если ни одна из групп, назначенных нодам, не содержит флексбоди.

У динамической коллизии есть ещё одно особое свойство. Если в радиусе более 10 см от треугольника нет ни одной ноды с включённой коллизией, треугольник становится неактивным с точки зрения обнаружения коллизий. Это помогает улучшить производительность, поэтому рекомендуется по умолчанию поддерживать расстояние более 10 см между коллидирующими компонентами там, где это возможно.

Силовая установка

Эти аргументы используются компонентами силовой установки автомобиля, такими как двигатель.

string
type
Особый вид группы, связанный с симуляцией силовой установки

Часто используемые группы двигателя:

  • engine_intake: Используется для симуляции повреждений от воды. Когда все ноды этой группы оказываются под водой, двигатель начинает заполняться водой — до тех пор, пока одна из нод не поднимется на поверхность, или двигатель не выйдет из строя от повреждений.
  • engine_block: Группа, которую разделяют все ноды, образующие структурный «куб» из Jbeam для двигателя автомобиля. Каждому двигателю должно принадлежать не менее 2 таких нод. Они используются тепловой симуляцией для расчёта массы компонентов двигателя и определения местоположения частичных эффектов при отказе двигателя.
  • radiator: Используется для определения местоположения и направления частиц дыма из радиатора. В этой engineGroup должно быть не менее 2 нод.
  • fuel: Эти ноды содержат массу топлива в топливном баке транспортного средства, которая равномерно распределяется между ними и прибавляется к их nodeWeight. На транспортном средстве может быть несколько топливных баков, каждый со своим именем engineGroup для топлива — fuel является наиболее распространённым. Другие ёмкостные баки на транспортном средстве также могут использовать эту систему — например, баки с закисью азота и баллоны с воздухом под давлением.

Отладка

Предназначены для кратковременной отладки и должны быть удалены из финальной версии конструкции.

dictionary
type
{"radius":0,"col":"#00000000"}
default
Подсвечивает ноду с указанным радиусом и цветом при включённом режиме отладочной визуализации нод.

Таблица состоит из 2 элементов: "radius" — число, задающее радиус подсветки в метрах, а "col" — либо строка с шестнадцатеричным значением цвета, либо таблица со значениями RGB. Пример использования:

["a", -0.90, -0.93, 0.23, {
    "highlight":{
        "radius":0.005,

            "r":255,
            "g":0,

        }
    }
