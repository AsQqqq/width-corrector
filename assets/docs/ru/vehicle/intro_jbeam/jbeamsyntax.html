Обзор

JBeam — это формат файла, определяющий физический скелет в движке BeamNG. Название JBeam происходит от того, что он основан на JSON (с некоторыми исключениями) для описания конструкций нод/балок.

Практически всё чувствительно к регистру, и формат совершенно не терпим к синтаксическим ошибкам, поэтому будьте внимательны! Файлы jbeam создаются и редактируются в текстовом редакторе (например, в Visual Studio Code или других редакторах с расширенными возможностями).

Отличия от JSON

Мы немного изменили парсер JSON , чтобы облегчить жизнь авторам транспортных средств:

  • Комментарии: поддерживаются комментарии в стиле C — многострочные и однострочные: //... и /* ... */
  • Запятые: все запятые необязательны, однако рекомендуется опускать их только в конце строк.

Общая структура 

{
  "vehicle": {
    "refNodes":[
        ["ref:", "back:", "left:", "up:"]
        ["f3r", "f5r", "f4l", "f8r"]
    ],
    "cameraExternal":{
      "distance": 6.7,
      "distanceMin": 9,
      "offset": {"x": 0.43, "y": 0.11, "z": 0.55},
      "fov": 77,
    },
  }
}

Основные концепции

Данные в файле jbeam можно разделить на два типа форматирования:

Таблицы

Представьте себе таблицу в электронной таблице: списки всегда имеют строку заголовка, определяющую имена столбцов, после которой следуют данные. Например:

{
  "vehicle": {
    "nodes": [ /* ... */ ],
    "refNodes":[
      ["ref:", "back:", "left:", "up:"],
      ["f3r", "f5r", "f4l", "f8r"],
      ["f2r", "f3r", "f1l", "f1r"]
    ]
  }
}

Таким образом, в приведённом примере f3r — это первое значение в своей строке в столбце ref: таблицы refNodes, которая содержится в vehicle. Представьте это следующим образом:

Таблицы используются тогда, когда данных одного типа много, что позволяет не повторять ключ снова и снова.

После разбора формат преобразуется в хранилище ключ–значение, и каждая строка сопоставляется с заголовками столбцов. Приведённый выше пример превращается для движка в следующее:

{
  "vehicle": {
    "nodes": [ /* ... */ ],
    "refNodes": [
      {
        "ref:nodes": "f3r",
        "back:nodes": "f5r",
        "left:nodes": "f4l",
        "up:nodes": "f8r",
      },
      {
        "ref:nodes": "f2r",
        "back:nodes": "f3r",
        "left:nodes": "f1l",
        "up:nodes": "f1r",
      }
    ]
  }
}

Ссылки на секции

Двоеточие (:) в строке заголовка таблицы задаёт ссылку на другую секцию транспортного средства. Формат следующий: valueName:targetSectionName. Если targetSectionName не указан, по умолчанию используется nodes.

Например, заголовок таблицы в строке 15:

 1{
 2  "vehicle": {
 3    "nodes": [
 4      ["id", "posX", "posY", "posZ"],
 5      ["f3r", -0.35, -1.56, 0.25],
 6      ["f5r",  0.00, -1.58, 0.24],
 7      ["f4l", -0.37, -0.98, 0.44],
 8      ["f8r", -0.37, -0.98, 0.22],
 9      ["f2r",  0.37, -0.98, 0.22],
10      ["f3r",  0.37, -0.98, 0.44],
11      ["f1l", -0.69, -0.62, 0.39],
12      ["f1r", -0.65, -0.64, 0.22]
13    ],
14    "refNodes":[
15      ["ref:", "back:", "left:", "up:"],
16      ["f3r", "f5r", "f4l", "f8r"],
17      ["f2r", "f3r", "f1l", "f1r"]
18    ]
19  }
20}

В приведённом примере при наличии "ref:nodes": "f3r" движок обратится к id = f3r в таблице nodes.

Словари

Это хранилища данных вида key-value. Они применяются тогда, когда данные достаточно уникальны и не повторяются.

{
  "vehicle": {
    "cameraExternal": {
      "distance": 6.7,
      "distanceMin": 9,
      "offset": {"x": 0.43, "y": 0.11, "z": 0.55},
      "fov": 77,
    },
  }
}

Словари не требуют постобработки, однако используются редко.

Внешние модификаторы (out-of-bounds)

Движок обрабатывает множество входных переменных, поэтому мы добавили способ изменять определённые значения.

Модификаторы области видимости

Например:

 1{
 2  "vehicle": {
 3    "nodes": [
 4      ["id", "posX", "posY", "posZ"],
 5      ["f3r", -0.35, -1.56, 0.25],
 6      ["f5r",  0.00, -1.58, 0.24],
 7      ["f4l", -0.37, -0.98, 0.44],
 8      ["f8r", -0.37, -0.98, 0.22],
 9      {"nodeWeight": 3},
10      ["f2r",  0.37, -0.98, 0.22],
11      ["f3r",  0.37, -0.98, 0.44],
12      ["f1l", -0.69, -0.62, 0.39],
13      ["f1r", -0.65, -0.64, 0.22]
14    ],
15  }
16}

Строка 9 в приведённом примере явно не является обычной строкой таблицы. (Любые строки должны быть обёрнуты в [ ]). Она добавит пару ключ–значение {"nodeWeight":3.0} ко всем последующим строкам на том же уровне и ниже (в той же области видимости).

Таким образом, нода f2r для движка будет выглядеть следующим образом:

1{
2  "id": "f2r",
3  "posX":  0.37,
4  "posY": -0.98,
5  "posZ":  0.22,
6  "nodeWeight": 3,
7}
Эта функция очень мощная, но одновременно и опасная: модификаторы области видимости могут «протечь» в другие файлы Jbeam, загружаемые после, что приведёт к непредсказуемым последствиям. Используйте с осторожностью.

Отменить действие модификаторов области видимости также возможно, установив им пустые значения:

 1{
 2  "vehicle": {
 3    "nodes": [
 4      ["id", "posX", "posY", "posZ"],
 5      ["f3r", -0.35, -1.56, 0.25],
 6      ["f5r",  0.00, -1.58, 0.24],
 7      {"group":"body"},
 8      ["f4l", -0.37, -0.98, 0.44],
 9      ["f8r", -0.37, -0.98, 0.22],
10      {"group":""},
11      ["f2r",  0.37, -0.98, 0.22],
12      ["f3r",  0.37, -0.98, 0.44],
13      ["f1l", -0.69, -0.62, 0.39],
14      ["f1r", -0.65, -0.64, 0.22]
15    ],
16  }
17}

В приведённом примере модификатор group будет применён только к нодам f4l и f8r.

Модификаторы строки таблицы

Например:

1{
2  "vehicle": {
3    "nodes": [
4      ["id", "posX", "posY", "posZ"],
5      ["f2r",  0.37, -0.98, 0.22, {"nodeWeight": 3}],
6      ["f3r",  0.37, -0.98, 0.44],
7    ],
8  }
9}

Строка 5 в приведённом примере содержит строку со словарём после столбцов. Эти значения применяются только к данной строке и нигде больше не «просачиваются». Нода f2r будет иметь свойство nodeWeight, нода f3r — нет.

Это предпочтительный способ задания дополнительных модификаторов, поскольку они не проникают в другие части ниже и не затрагивают их.

Анатомия файла jbeam транспортного средства

Транспортные средства в BeamNG собираются из деталей; детали содержат так называемые sections (секции), хранящие фактические данные.

Каждый файл jbeam представляет собой плоский словарь ключ–значение доступных деталей, содержащихся в этом файле. Например:

1{
2"autobello_fender_RR": {
3  // ...
4},
5"autobello_fender_RL": {
6  // ...
7},
8}

Ключ — это имя детали, его значение — её секции. При загрузке движок строит дерево деталей и должен знать, какая деталь является base (базовой) или root (корневой). За это отвечает "slotType":"main".

Продвинутые возможности

Ниже описаны некоторые расширенные части синтаксиса. Они могут быть полезны опытному моддеру, но могут показаться сложными для новичка.

Функции

Функции используются для динамического вычисления числовых значений в Jbeam (например, когда число зависит от переменной), а также для конкатенации строковых значений. По сути это автоматически вычисляемые математические и/или логические выражения. Несмотря на ограниченное применение в официальном контенте, они обладают очень широкими возможностями.

Поскольку файлы Jbeam по сути являются базами данных изолированных (sandboxed) lua-выражений, можно строить практически любые выражения, допустимые в чистом lua. Единственное исключение — в Jbeam нельзя присваивать новые lua-переменные, поэтому результаты вычислений, сделанных в более ранних секциях Jbeam, повторно использовать не получится.

Чтобы заменить числовое значение функцией, нужно заключить его в двойные кавычки, как строку, и начать выражение с $=. Например:

{
  "vehicle": {
    "nodes": [
      {"nodeWeight": 4}, //before
      {"nodeWeight": "$=2+2"}, //after
    ],
  }
}

Конкатенация строки аналогична: двойные кавычки должны охватывать всю функцию целиком, а все подстроки следует заключать в одинарные кавычки:

{
  "vehicle": {
    "nodes": [
      {"group": "my_group"}, //before
      {"group": "$='my_'..'group'"}, //after
    ],
  }
}

Функции тесно связаны с переменными и часто используются совместно с ними. Переменная, используемая самостоятельно, должна быть в двойных кавычках, но при использовании внутри функции её собственные кавычки выполняют эту роль.

{
  "vehicle": {
    "nodes": [
      {"nodeWeight": "$weight"}, //before
      {"nodeWeight": "$=$weight*1"}, //after
    ],
  }
}

Функции можно использовать везде в Jbeam, за исключением аргумента 'id' секции rails — там нужно использовать rails2 , которая поддерживает это. Rails2 — это по сути rails с другим синтаксисом, поэтому преобразование всегда выполняется без потерь.

Встроенные функции

Несколько lua-файлов, входящих в состав игры, предоставляют готовые функции для использования в файлах Jbeam. Импортировать библиотеки или добавлять префиксы не нужно — они доступны сразу. Их можно использовать как самостоятельно, так и в составе собственных пользовательских функций, применяя к числам, переменным или другим функциям. Хотя большинство из них редко встречается в официальном контенте, вы можете свободно применять их в модах.

Продвинутые модификаторы

Некоторые расширенные внешние модификаторы, которые могут встречаться в синтаксисе Jbeam. Они используются только в особых случаях.

Модификаторы масштабирования

В основной части Jbeam, рядом с таблицами и словарями, существует особый вид модификаторов. Он используется для масштабирования числовых значений других модификаторов.

 1{
 2  "vehicle": {
 3    "slots": [
 4        ["type", "default", "description"],
 5        ["vehicle_body","vehicle_body", "Body"],
 6        ["paint_design","", "Paint Design"],
 7    ],
 8    "scaledragCoef":2.15,
 9    "controller": [
10        ["fileName"],
11        ["vehicleController", {}],
12    ],
13  }
14}

Модификатор работает следующим образом: он ищет строку scale в ключах внутри структуры Jbeam рядом с таблицами и словарями, а затем умножает все модификаторы с именем, указанным в остатке строки, на заданное значение. Например, приведённый выше модификатор умножит все модификаторы dragCoef на 2.15. Он действует глобально на всё транспортное средство, пока не будет загружена деталь, которая его отменяет.

Чаще всего он применяется для точной настройки общего коэффициента аэродинамического сопротивления транспортного средства, однако с его помощью можно масштабировать и любые другие модификаторы. Имейте в виду, что это затрудняет последующее чтение файла Jbeam, поскольку вы больше не работаете с реальными значениями, поэтому использовать его следует только в особых случаях.

Модификатор отключения (Disable)

Модификатор Disable — это специальный модификатор, который заставляет lua пропускать строки и не читать их. Например, следующие строки не будут прочитаны, а строки ниже — будут:

[
  {"disable": true},

  ["5","7","6"],
  ["5","6","4"],

  ["0","5","4"],

  ["2","6","7"],
  ["2","7","3"],
  {"disable": ""},
]

Он полезен в сочетании с переменными слотов (Slot Variables) , передающими булево значение. Это позволяет объединить несколько вариантов детали Jbeam с небольшими отличиями, имеющих отдельные slotType, в одну деталь: секции, содержащие изменения, окружаются модификаторами Disable, использующими булевы переменные слотов:

[
  {"disable": "$variant1"},

  ["5","7","6"],
  ["5","6","4"],

  ["0","5","4"],

  ["2","6","7"],
  ["2","7","3"],

  {"disable": "$variant2"},

  ["5b","7","6"],
  ["5b","6","4"],

  ["0","5b","4"],

  ["2b","6","7"],
  ["2b","7","3"],

  {"disable": ""},
]

Модификатор включения (Include)

Доступен начиная с версии BeamNG 0.34; модификатор Include заменяет устаревший и не рекомендуемый к использованию модификатор includeCSV. Его назначение — хранить содержимое таблицы Jbeam в CSV-файле, путь к которому задаётся относительно папки игры. Для этого используется синтаксис выражений, аналогичный переменным:

{
  "mainEngine":{
      "torque": "$=include('vehicles/common/engines/v8_classic/gavril_291_mainTorqueCurve.csv')",
      "idleRPM":700,
      //...
  }
}

Как игра видит эту секцию:

      "torque":[
          ["rpm", "torque"],
          [0,    0],
          [500,170],
          [1000,283],
          [1500,342],
          [2000,370],
          [2500,390],
          [3000,402],
          [3500,403],

          [7500,98],
          [8000,65],

Этот модификатор удобен для хранения общих данных Jbeam — в особенности компонентов трансмиссии — совместно используемых разными транспортными средствами. Например, два автомобиля могут использовать один и тот же двигатель, но не могут использовать один базовый файл Jbeam из-за немного различающихся флексбоди, скорректированных под разные зазоры, положения воздухозаборников и т.д.

Помимо CSV-файлов, поддерживаются также файлы XLSX. Имейте в виду, что парсер XLSX работает медленно и должен использоваться только для прототипирования и отладки — в финальном продукте следует использовать CSV. Пример использования XLSX, файл доступен здесь :

{
  "mainEngine":{
      "torque": "$=include('vehicles/common/engines/v8_classic/gavril_291_mainTorqueCurve.xlsx')",
      "idleRPM":700,
      //...
  }
}

Если файл XLSX содержит несколько листов, можно указать нужный во втором аргументе модификатора:

{
"torque": "$=include('vehicles/common/engines/v8_classic/gavril_291_mainTorqueCurve.xlsx', 'Sheet1')",
}

При указании листа можно дополнительно сузить диапазон до конкретных ячеек с помощью третьего аргумента. Библиотека поддерживает различные виды запросов диапазона:

  • Одна ячейка: укажите одну ячейку, например "B2".
  • Весь столбец: укажите столбец, например "C".
  • Вся строка: укажите строку, например "3".
  • Диапазон столбцов: укажите диапазон столбцов, например "B:D".
  • Диапазон строк: укажите диапазон строк, например "2:4".
  • Диапазон ячеек: укажите прямоугольный диапазон, например "A1:C3".
  • Весь лист: не указывайте диапазон, чтобы получить весь лист.

Пример использования:

{
"torque": "$=include('vehicles/common/engines/v8_classic/gavril_291_mainTorqueCurve.xlsx', 'Sheet1', 'A1:C3')",
}

Эти аргументы доступны только для файлов XLSX и не поддерживаются для CSV.

Переменные пространства имён

Переменные пространства имён — это разновидность переменных слотов (Slot Variables) , добавленная в версии 0.38. Это 2 специальные переменные с именами $prefix и $suffix.

В дочерней детали, которой переданы эти переменные, любая строка, начинающаяся с конструкции $., будет иметь переменную $prefix, добавленную в начало, и $suffix — в конец. Если из родительского слота передана только одна из этих переменных, добавлена будет только она. Если переданы обе — добавятся обе.

Например, можно передать в дочернюю деталь следующие переменные пространства имён:

"variables":{"$prefix":"row1_", "$suffix":"_part1"}

Затем в дочерней детали можно добавить ноду следующим образом:

["$.node1", 0, 1, 0],

Тогда имя ноды будет прочитано игровым движком как row1_node1_part1.

Переменные пространства имён существуют для сокращения синтаксиса при активном использовании переменных слотов в различных секциях Jbeam. Запись "$.node1" по существу эквивалентна записи:

"$= ($prefix ~= nil and $prefix or '') .. 'node1' .. ($suffix ~= nil and $suffix or '')"

Это даёт значительную экономию времени и позволяет навести порядок в случаях, когда одну и ту же деталь Jbeam нужно использовать в нескольких слотах — это вынуждает делать имена нод зависящими от переменных слотов и ссылаться на динамические имена во всех секциях, таких как beams и triangles.
