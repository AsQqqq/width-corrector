версия 31 / 0x001F

В этом документе описывается формат файла v31 BeamNG cached Collada (.cdae), создаваемый внутри кодовой базы BeamNG. Данные сериализуются в формате MessagePack с возможным сжатием Zstandard. Используйте эту информацию для создания или отладки собственных инструментов чтения файла.

Почему используется бинарный формат: скорость и эффективность

Файл .cdae хранит данные формы в бинарном формате по нескольким причинам:

  1. Меньший размер: Бинарные данные имеют меньше накладных расходов по сравнению с текстовыми форматами, что снижает использование дискового пространства и время загрузки.
  2. Более быстрый разбор: Инструменты и движок могут значительно быстрее считывать целые числа, числа с плавающей точкой и другие элементы в бинарном виде, нежели разбирать их из текста.
  3. Упрощённый рабочий процесс: В бинарном формате расположение данных (например, float, int, векторы) ближе к структурам памяти, что сводит к минимуму количество преобразований.
  4. Единообразное представление: Бинарный формат исключает неоднозначность в форматировании, делая операции чтения/записи более надёжными.

Этот компактный и последовательный подход в конечном счёте повышает производительность при загрузке форм, особенно в масштабе.

Обзор содержимого файла .cdae (v31)

Формат v31 вводит более структурированный подход с отдельным заголовком и телом, а также опциональным сжатием:

Приблизительная псевдокодовая таблица содержимого файла .cdae версии v31

Шаг Содержимое Примечания
1 Заголовок файла (32-битное целое, версия). Проверьте version & 0xFF == 31, чтобы убедиться в версии v31.
2 Размер заголовка (32-битное беззнаковое целое). Размер следующих данных заголовка в байтах.
3 Заголовок MsgPack (словарь с метаданными) Содержит информацию о флаге сжатия, размере тела и т. д.
4 Данные тела (возможно, сжатые) При наличии сжатия используйте Zstandard для распаковки. Содержит данные формы.

Заголовок — это словарь MsgPack, содержащий метаданные о файле, включая информацию о том, сжато ли тело. Тело содержит фактические данные формы также в формате MsgPack.

Структура файла: заголовок и тело

В начале файла формат начинается со следующего:

  1. 32-битное целое version:

    version = (smVersion | (mExporterVersion << 16))

    где smVersion установлен в 31 для данного формата, а exporter_version — отдельное 16-битное значение в старших битах.

    Читателю следует проверить (version & 0xFF) == 31, чтобы убедиться, что это форма v31.

  2. 32-битное беззнаковое целое headerSize: Указывает размер следующих данных заголовка MsgPack.

  3. Заголовок MsgPack: Словарь, содержащий метаданные о файле, такие как:

    • compression: Булево значение, указывающее, сжато ли тело с помощью Zstandard
    • bodysize: Размер данных тела (в байтах)
    • Прочие возможные метаданные

  4. Данные тела:

    • Если compression равно true, тело сжато с помощью Zstandard и требует распаковки
    • После распаковки (при необходимости) тело содержит фактические данные формы в формате MsgPack

Обзор данных тела MsgPack

После считывания и (при необходимости) распаковки данных тела можно разобрать объекты MsgPack в определённом порядке:

  1. Информация о форме:

    • mSmallestVisibleSize (float)
    • mSmallestVisibleDL (int32)
    • radius (float)
    • tubeRadius (float)
    • center (Point3F — 3 float-значения)
    • bounds (Box3F — 6 float-значений)

  2. Несколько векторов (pack_vector), каждый хранится как:

    • 32-битное знаковое целое для длины вектора
    • 32-битное знаковое целое для размера элемента в байтах
    • Блок необработанных бинарных данных (MsgPack «bin») длиной vector.size * elementSize.

    Код считывает их в следующем порядке:

    • nodes

    • objects

    • subShapeFirstNode

    • subShapeFirstObject

    • subShapeNumNodes

    • subShapeNumObjects

    • defaultRotations

    • defaultTranslations

    • nodeRotations

    • nodeTranslations

    • nodeUniformScales

    • nodeAlignedScales

    • nodeArbitraryScaleFactors

    • nodeArbitraryScaleRots

    • groundTranslations

    • groundRotations

    • objectStates

    • triggers

    • details

  3. Массив строк

    • Значение uint32 для количества имён формы.
    • Каждое имя — строка MsgPack.

  4. Меши

    • Значение uint32 — общее количество мешей.
    • Каждый меш содержит:

      • uint32 meshType:

        • 0 → StandardMesh
        • 1 → SkinMesh
        • 2 → DecalMesh (устарело)
        • 3 → SortedMesh
        • 4 → NullMesh

      • Для ненулевых мешей:

        • int32 numFrames
        • int32 numMatFrames
        • int32 parentMesh
        • Box3F mBounds (6 float-значений)
        • Point3F mCenter (3 float-значения)
        • float mRadius

        Затем дополнительные вызовы pack_vector для:

        • verts
        • tverts
        • tverts2
        • colors
        • norms
        • encodedNorms
        • primitives
        • indices
        • tangents

        И в конце:

        • int32 vertsPerFrame
        • uint32 meshFlags

      • Для мешей типа SkinMesh (тип 1) считываются дополнительные данные:

        • Сначала определяется, является ли меш родительским (parentMesh < 0)
        • Для родительских мешей:
          • Вектор skinMesh.initialVerts
          • Вектор skinMesh.initialNorms
        • Для всех мешей типа SkinMesh:
          • Вектор skinMesh.initialTransforms
        • Для родительских мешей:
          • Вектор skinMesh.vertexIndex
          • Вектор skinMesh.boneIndex
          • Вектор skinMesh.weight
          • Вектор skinMesh.nodeIndex

  5. Последовательности (Sequences)

    • Значение uint32 numSequences.
    • Для каждой последовательности:
      • int32 nameIndex
      • uint32 flags
      • int32 numKeyframes
      • float duration
      • int32 priority
      • int32 firstGroundFrame
      • int32 numGroundFrames
      • int32 baseRotation
      • int32 baseTranslation
      • int32 baseScale
      • int32 baseObjectState
      • int32 baseDecalState
      • int32 firstTrigger
      • int32 numTriggers
      • float toolBegin
      • TSIntegerSet rotationMatters
      • TSIntegerSet translationMatters
      • TSIntegerSet scaleMatters
      • TSIntegerSet visMatters
      • TSIntegerSet frameMatters
      • TSIntegerSet matFrameMatters

  6. Список материалов (необязательно)

    • Значение uint32 materialCount.
    • Для каждого материала:
      • string name
      • uint32 flags
      • uint32 reflectanceMap
      • uint32 bumpMap
      • uint32 detailMap
      • float detailScale
      • float reflectionAmount

На этом данные заканчиваются. Краткие определения:

  • Point3F: 3 float-значения [x, y, z].
  • Box3F: минимальный и максимальный углы => 6 float-значений итого.
  • TSIntegerSet: специализированный битовый набор для флагов узлов/элементов.

Стратегия чтения (псевдокод)

  1. Считайте первые 4 байта (32-битное целое) → version.

    • Проверьте version & 0xFF == 31.
    • Извлеките exporter_version = version >> 16.

  2. Считайте следующие 4 байта (32-битное беззнаковое целое) → headerSize.

  3. Считайте headerSize байт и разберите как MsgPack → header_info.

    • Проверьте, является ли header_info['compression'] истинным.
    • Запомните значение bodysize.

  4. Считайте оставшиеся байты → body_data.

    • Если сжатие включено, распакуйте с помощью Zstandard.

  5. Передайте распакованное тело в парсер MsgPack.

  6. Считайте информацию о форме:

    • mSmallestVisibleSize (float)
    • mSmallestVisibleDL (int32)
    • radius (float)
    • tubeRadius (float)
    • center (3 float-значения)
    • bounds (6 float-значений)

  7. Считайте несколько векторов в точном порядке, описанном выше.

  8. Считайте массив имён: количество (uint32), затем столько же строк.

  9. Считайте меши: общее количество (uint32), затем данные каждого меша.

    • Для мешей типа SkinMesh обрабатывайте родительские и дочерние меши по-разному, с дополнительными векторами.

  10. Считайте последовательности: количество (uint32), затем данные каждой последовательности.

  11. Считайте список материалов (при наличии): количество (uint32), затем данные каждого материала.
