Виртуальная машина — это каждая из сред, в которых может выполняться код. Существует 3 типа виртуальных машин:

  • Vehicle Lua VM (VLUA): выполняет код, специфичный для конкретного транспортного средства (например, расчёт наддува турбины или входные данные рулевого управления ИИ).
  • GameEngine Lua VM (GELUA): выполняет общий код, не привязанный к конкретному транспортному средству (например, карьерный режим, управление трафиком или воспроизведение записей).
  • User Interface VM (UI): выполняет javascript/html для формирования визуального интерфейса (например, главного меню).

VM GELUA и Main Menu UI существуют всегда; остальные же VM зависят от используемых транспортных средств (а в некоторых случаях — и от карт или модов в целом).

Пример

Представьте, что вы заспаунили современное такси и хэтчбек 90-х годов. Тогда у вас могут быть следующие VM:

  • Общие VM:
    • GELUA
    • Main Menu UI
  • VM такси:
    • VLUA
    • Dashboard UI
    • GPS navigator UI
    • Infotainment UI
  • VM хэтчбека:
    • VLUA

Итого — 7 различных VM, работающих параллельно.

Расположение исходного кода

Структура папок:

  • /ui/: код VM пользовательского интерфейса (javascript и html).
  • /lua/vehicle/: код VM VLUA.
  • /lua/ge/: код VM GELUA.
  • /lua/common/: библиотеки, доступные как в VLUA, так и в GELUA.

Библиотеки из /lua/common/ — удобный способ избежать дублирования кода, однако они не являются средством совместного использования или передачи информации между различными VM. Если вам нужно обмениваться данными, обратитесь к разделу communication.

Многопоточность

Каждая VM работает в собственной изолированной среде:

  • VM пользовательского интерфейса: каждая выполняется в отдельном процессе.
  • VM VLUA: каждая выполняется в отдельном потоке.
  • VM GELUA: выполняется в главном потоке.

Частота обновления

Каждая VM может обновляться с разной частотой в зависимости от своего назначения. В качестве общего правила:

  • VLUA: может работать с частотой физического движка (2000 Гц, см. lua/vehicle/main.lua::onPhysicsStep()) или с частотой графики (см. lua/vehicle/main.lua::onGraphicsStep()).
  • GELUA: может работать с частотой графики (текущий «фреймрейт», см. lua/ge/main.lua::update()) или с частотой UI (см. событие lua/ge/main.lua::onGuiUpdate).
  • Main UI: работает с переменной частотой. В идеале — 60 FPS, но часто 30 FPS и ниже. Это зависит от доступных ресурсов компьютера.
  • Secondary UIs: работают с переменной частотой, как правило более низкой, чем у Main UI.

Помимо этого, вы можете реализовать собственную фиксированную частоту обновления. Один из способов — привязать свой код к вызову графического update(), а затем вручную пропускать тики на основе простого отслеживания времени. Таким образом, ваш код будет выполняться не чаще, чем с частотой графики, как правило — с фиксированной частотой, которую вы задали. Тем не менее мы не рекомендуем использовать логику с фиксированной частотой из соображений производительности. Подробнее — по ссылке ниже.

Примечание: мы понимаем, что писать математику, корректно работающую при экстремальных колебаниях частоты обновления, непросто. По этой причине гарантируется, что сообщаемая частота графического обновления будет не ниже 20 Гц. Если компьютер не способен выдать 20 FPS, симуляция будет замедляться по мере необходимости. Это гарантия, позволяющая программировать математику с безопасной базовой частотой.

Выбор правильной частоты обновления для вашего кода крайне важен для того, чтобы мод корректно работал на различных компьютерах. Подробнее см. в разделе Improving Framerate.

Перезагрузка

В процессе разработки мода вам может потребоваться изменить исходный код, выполняемый в конкретной VM. Чтобы ускорить итерации, BeamNG позволяет перезагружать нужную VM на лету — это даёт возможность сразу проверить новые изменения без перезапуска программы:

  • Main UI: нажмите F5.
  • GELUA: нажмите Ctrl-L.
  • VLUA: переключитесь на нужное транспортное средство клавишей Tab, затем нажмите Ctrl-R.
  • VLUA (все): нажмите Ctrl-Shift-R.

При таких перезагрузках всё начинается с чистого листа: полностью новая VM создаётся с нуля, а старая удаляется. Тем не менее в BeamNG есть удобная функция: система попытается восстановить/продолжить работу с того места, где вы остановились, — для этого сохраняется старое состояние и загружается в новую VM после её запуска.

Идея в том, чтобы продолжить разработку именно с того момента, на котором вы остановились, — а не начинать заново, например загружая сохранение карьерного режима после каждого редактирования кода.

Взаимодействие

Каждая виртуальная машина работает параллельно остальным и полностью изолирована от них.

Если вам нужно, чтобы информация из одной VM стала доступна в другой, необходимо явно передать эту информацию из одной VM в другую.

Мы предлагаем следующие асинхронные методы взаимодействия:

  • Асинхронные очереди в VLUA и GELUA: вы помещаете в очередь строку исходного кода, адресованную целевой LUA VM. Эта строка содержит нужную информацию. Например, в VM VLUA можно поставить в очередь строку "input.event('throttle', 0.6)".
  • Асинхронные события в UI: вы ставите в очередь событие, о котором будет уведомлён javascript, и различные слушатели могут на него реагировать. Например, можно поставить в очередь событие "MenuHide".
  • Асинхронные почтовые ящики в VLUA: VM VLUA также могут считывать данные из системы виртуальных почтовых ящиков: отправитель записывает данные в ящик, а получатель(и) могут проверить, какая информация последней туда поступила. Это более производительный способ взаимодействия по сравнению с очередями, однако выполнять произвольный код таким образом нельзя — принимающая VM должна самостоятельно проверить свой почтовый ящик, чтобы обнаружить обновлённые данные, и обработать их по своему усмотрению.

Примечание: если вы задавались вопросом о традиционных методах взаимодействия (таких как мьютекс, барьер…), они намеренно не предоставляются — ради надёжности и простоты.

А вот та же информация с наглядными примерами кода:

  • Отправка событий в UI:
    • guihooks.trigger("myEvent") (из VLUA)
    • guihooks.trigger("myEvent") (из GELUA)
    • $scope.$emit("myEvent") (из UI)
  • Прослушивание событий в UI:
    • $scope.$on("myEvent", function() { myJavascriptCode() } )
  • Постановка кода в очередь для GELUA:
    • bngApi.engineLua('myLuaCode()') (из UI)
    • obj:queueGameEngineLua('myLuaCode()') (из VLUA)
  • Постановка кода в очередь для конкретной VLUA:
    • bngApi.engineLua("be:getPlayerVehicle(0):queueLuaCommand('myLuaCode()')") (из UI)
    • be:getPlayerVehicle(0):queueLuaCommand('myLuaCode()') (из GELUA)
  • Постановка кода в очередь для всех VLUA:
    • bngApi.engineLua("be:queueAllObjectLua('myLuaCode()')") (из UI)
    • be:queueAllObjectLua('myLuaCode()') (из GELUA)
    • BeamEngine:queueAllObjectLua('myLuaCode()') (из VLUA)
  • Постановка кода в очередь для всех VLUA, кроме одной:
    • BeamEngine:queueAllObjectLuaExcept('myLuaCode()', exceptObjectID) (из VLUA)
  • Запись информации в почтовый ящик VLUA:
    • be:sendToMailbox("myMailboxAddress", myData) (из GELUA)
  • Чтение информации из почтового ящика VLUA:
    • local myData = obj:getLastMailbox("myMailboxAddress")

Примечание: чтение данных из почтового ящика не удаляет информацию. Она по-прежнему доступна для последующего чтения любой другой VM VLUA.

Рекомендуем изучить существующий официальный код — это поможет лучше понять возможности системы и увидеть рабочие примеры вместо псевдокода, приведённого выше.

Полная архитектура

Разделы выше охватывают все основы, необходимые для программирования модов.

Однако если вам нужны дополнительные подробности, существуют дополнительные компоненты, связывающие все эти VM воедино:

  • C++ Game Engine: «настоящий» высокопроизводительный движок, написанный преимущественно на C++ и реализующий такие возможности, как рендеринг кадров, обработка ввода, VR, генерация звука и т. д.
  • C++ Physics Core: в то время как VM VLUA вычисляют вспомогательные части физики, связанные с транспортными средствами (например, трансмиссию и т. п.), существует также ядро физического движка. Оно вычисляет физику мягкого тела, модели трения, физику захвата нод и т. д., а также управляет выполнением и синхронизацией всех VM VLUA.
  • TS VM: этот компонент ещё не удалён полностью, поэтому в нашем движке сохранились остатки старой VM TorqueScript. Она работает в главном потоке и не должна использоваться в новых разработках или модах.

Однако для целей моддинга всё это можно игнорировать — достаточно сосредоточиться только на VM UI/VLUA/GELUA.
