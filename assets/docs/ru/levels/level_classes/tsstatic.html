TSStatic — основной класс объектов для размещения отдельных статических мешей в уровнях BeamNG. Он широко используется для зданий, знаков, мостов и других вручную расставленных элементов окружения.

Для очень большого количества повторяющихся статических объектов — деревьев, камней, кустов, мелкой растительности, реквизита, отбойников — система Forest обычно эффективнее. Элементы Forest являются предпочтительным лёгким решением для повторяющихся ассетов, тогда как TSStatic лучше подходит для индивидуально размещённых объектов.

Базовый пример

Простая запись TSStatic в items.level.json:

{
  "class": "TSStatic",
  "name": "garage_building",
  "shapeName": "/levels/example/art/shapes/garage.dae",
  "position": [100, 200, 0],
  "rotationMatrix": [1, 0, 0, 0, 1, 0, 0, 0, 1],
  "scale": [1, 1, 1],
  "collisionType": "Collision Mesh",
  "decalType": "Collision Mesh",
  "isRenderEnabled": true
}

Важные поля

Поле Тип Описание
class string Должно быть "TSStatic".
name string Имя объекта сцены. Должно быть уникальным в пределах уровня.
shapeName string Путь к файлу модели, используемой данным объектом.
position array[3] Мировая позиция.
rotationMatrix array[9] Вращение объекта в виде матрицы 3x3.
scale array[3] Масштаб объекта.
collisionType enum/string Данные меша, используемые для физики и запросов коллизий.
decalType enum/string Данные меша для проецирования и отсечения декалей.
dynamic bool Отключает некоторые оптимизации рендеринга статических объектов.
playAmbient bool Воспроизводит анимационную последовательность с именем ambient, если она есть.
meshCulling bool Включает пофрагментное отсечение для больших сложных форм.
originSort bool Сортирует полупрозрачный рендеринг по началу координат объекта вместо границ.
useInstanceRenderData bool Включает данные рендеринга на уровне экземпляра, например цвета экземпляров.
instanceColor color Множитель базового цвета для отдельного экземпляра.
instanceColor1 color Первый цвет палитры для материалов с цветовой палитрой экземпляров.
instanceColor2 color Второй цвет палитры для материалов с цветовой палитрой экземпляров.
skin string Необязательное переназначение целей материала / имя скина.
prebuildCollisionData bool Предварительно строит данные коллизий при загрузке.
renderNormals number Масштаб отладочного отображения нормалей.
forceDetail integer Принудительно задаёт конкретный уровень детализации/LOD формы. -1 означает автоматический выбор.

Файл формы

Самое важное поле:

"shapeName": "/levels/example/art/shapes/garage.dae"

Оно указывает на модель, используемую объектом.

Поддерживаемые пути к формам обычно указывают на:

.dae
.cdae
.dts

Движок может использовать кешированные данные форм внутри, например .cdae, но файлы уровней обычно ссылаются на исходный/импортированный путь к форме.

Если форму не удаётся загрузить, движок может заменить её на форму с ошибкой/без меша и записать ошибку в лог.

Поля трансформации

Большинство объектов TSStatic используют стандартные поля трансформации.

position 

"position": [100, 200, 0]

Мировая позиция в метрах.

rotationMatrix 

"rotationMatrix": [
  1, 0, 0,
  0, 1, 0,
  0, 0, 1
]

Матрица вращения 3x3.

scale 

"scale": [1, 1, 1]

Масштаб объекта.

Масштабирование влияет на рендеринг и коллизии. Большой неравномерный масштаб может затруднить проверку корректности коллизий и внешнего вида.

Тип коллизии

collisionType определяет, какие данные меша используются для физических коллизий и рейкастов.

Поддерживаемые значения:

Значение Описание
None Коллизия отсутствует.
Bounds Использует ограничивающий параллелепипед объекта. Ограниченная поддержка.
Collision Mesh Использует выделенные меши коллизий из формы. Рекомендуется.
Visible Mesh Использует полигоны визуального меша для коллизий. Ресурсоёмко.
Visible Mesh Final Намеренно использует полигоны визуального меша. Применяйте только при необходимости.

Пример:

"collisionType": "Collision Mesh"

Рекомендуемое значение по умолчанию:

"collisionType": "Collision Mesh"

Используйте None для чисто декоративных объектов, которые не должны участвовать в коллизиях:

"collisionType": "None"

Меши коллизий

Для наилучшей производительности модели должны включать выделенную геометрию коллизий.

Выделенный меш коллизий должен быть значительно проще, чем видимый меш.

Хорошие меши коллизий:

  • Низкополигональные
  • Достаточно простые для физики
  • Достаточно точные для геймплея
  • Без излишней детализации
  • Логически разделённые, если объект сложный

Избегайте использования коллизий на основе видимого меша, если это не необходимо.

Коллизия по видимому мешу может быть ресурсоёмкой. Если это намеренно, используйте Visible Mesh Final; в противном случае предпочтительны полноценные меши коллизий.

Тип декали

decalType определяет, по каким данным меша декали обрезаются/проецируются.

Пример:

"decalType": "Collision Mesh"

Допустимые значения те же, что и для collisionType.

Для объектов, на которые должны проецироваться декали, используйте тип меша, соответствующий поверхности.

Для объектов, которые не должны получать декали:

"decalType": "None"

Оптимизация рендеринга статических объектов

По умолчанию объекты TSStatic без анимации могут отрисовываться через менеджер статических форм. Это позволяет движку группировать и оптимизировать повторяющиеся статические формы.

Это важно для производительности, особенно когда уровень содержит большое количество расставленных мешей.

Объект может использовать рендеринг через статический менеджер при выполнении условий:

  • dynamic имеет значение false
  • Анимация ambient не активна
  • Форма может рассматриваться как статическая

Если объект динамический или анимированный, он использует собственный TSShapeInstance.

dynamic 

"dynamic": true

dynamic отключает некоторые оптимизации рендеринга статических объектов.

Используйте это только при необходимости — например, для объектов, которым требуется динамическое поведение на уровне объекта или которые не могут управляться как статический экземпляр.

Для обычного реквизита и зданий уровня оставьте значение false или опустите это поле.

Фоновая анимация

TSStatic может автоматически воспроизводить анимационную последовательность с именем:

ambient

если форма содержит её и включён параметр playAmbient.

"playAmbient": true

Это удобно для простых повторяющихся анимаций, например:

  • Вентиляторы
  • Вращающиеся знаки
  • Мелкие механические элементы
  • Анимированные элементы заднего фона

Если существует фоновая анимация, объект становится обрабатываемым и не будет использовать наиболее оптимизированный статический путь.

meshCulling 

"meshCulling": true

meshCulling включает более детальное попрефрагментное отсечение внутри объекта.

Это устаревшая функция, которая работает только при включённом dynamic; использовать её не рекомендуется, так как она создаёт проблемы с производительностью.

originSort 

"originSort": true

originSort изменяет поведение сортировки полупрозрачных объектов: вместо границ используется начало координат объекта.

Это может помочь с некоторыми прозрачными объектами, однако может также вызывать артефакты сортировки, если начало координат расположено неудачно.

Используйте только при необходимости для полупрозрачных материалов.

Данные рендеринга экземпляра

useInstanceRenderData включает свойства рендеринга на уровне отдельного экземпляра.

"useInstanceRenderData": true

Используется материалами, поддерживающими цвет экземпляра или данные палитры.

Важные поля:

"instanceColor": [1, 1, 1, 1],
"instanceColor1": [1, 0, 0, 1],
"instanceColor2": [0, 0, 1, 1]

Они могут управлять такими свойствами материала, как:

  • Базовый цвет экземпляра
  • Прозрачность экземпляра
  • Свечение экземпляра
  • Цветовые палитры

Материал должен быть создан с поддержкой этих функций.

Пример: реквизит с цветом экземпляра 

{
  "class": "TSStatic",
  "name": "colored_barrier",
  "shapeName": "/levels/example/art/shapes/barrier.dae",
  "position": [0, 0, 0],
  "rotationMatrix": [1, 0, 0, 0, 1, 0, 0, 0, 1],
  "scale": [1, 1, 1],
  "useInstanceRenderData": true,
  "instanceColor": [1, 0.2, 0.1, 1],
  "collisionType": "Collision Mesh"
}

Skin

Поле skin переназначает целевые материалы на форме.

"skin": "blue"

Базовое поведение:

  • Цели материалов, начинающиеся со старого имени скина, переименовываются.
  • Старое имя скина по умолчанию обычно равно base.
  • Новая цель должна указывать на существующий материал.

Пример:

base_body → blue_body

Расширенный синтаксис:

"skin": "base=blue;glass=darkglass"

Это заменяет несколько префиксов целей материалов.

Скиннинг изменяет имена целей материалов. Геометрия мешей при этом не изменяется.

Материалы

Цели материалов TSStatic берутся из списка материалов формы. Эти цели сопоставляются с определениями BeamNG Material через mapTo.

Например, если в модели есть цель материала:

garage_wall

то определение материала должно использовать:

"mapTo": "garage_wall"

Используйте редактор материалов или инструменты для проверки целей материалов.

Полезные методы времени выполнения для инструментов:

getTargetCount()
getTargetName(index)

LOD и уровни детализации

TSStatic использует уровни детализации формы для LOD.

Движок выбирает уровень детализации на основе расчётного размера объекта в пространстве экрана. Проще говоря, он определяет, насколько крупным объект выглядит на экране, и выбирает подходящий LOD.

На это влияет несколько факторов:

  • Расстояние до камеры — дальние объекты выглядят меньше, поэтому используются более низкие уровни детализации.
  • FOV камеры — угол обзора камеры влияет на то, насколько крупными или мелкими объекты выглядят на экране.
  • Размер формы — крупные объекты дольше остаются визуально большими, поэтому они сохраняют более высокие уровни детализации на больших расстояниях.
  • Масштаб объекта — увеличенные объекты ведут себя как более крупные и позже переключаются на более низкие LOD.
  • Размер вьюпорта / проекция — разрешение экрана и проекция камеры влияют на то, сколько пикселей занимает объект.
  • Глобальная настройка детализации — $pref::TS::detailAdjust масштабирует вычисленный размер в пикселях, позволяя повышать или понижать смещение LOD.

Результатом является вычисленное значение pixelSize. Оно сравнивается с уровнями детализации, хранящимися в форме. Большее значение pixelSize означает, что объект выглядит крупнее на экране и должен использовать меш с более высокой детализацией; меньшее значение pixelSize означает, что объект выглядит мельче и может использовать меш с более низкой детализацией.

Например, маленький реквизит и большое здание на одном расстоянии не обязательно будут использовать один и тот же LOD. У здания больший радиус ограничивающего объёма, оно занимает больше пространства на экране и может сохранять более высокий уровень детализации на большем расстоянии.

Соответствующие настройки:

$pref::TS::detailAdjust
$pref::TS::skipRenderDLs

forceDetail 

"forceDetail": -1

-1 означает автоматический выбор детализации.

Неотрицательное значение принудительно задаёт конкретный уровень детализации:

"forceDetail": 0

Используйте это только для отладки или особых случаев. Для обычного контента позвольте движку выбирать LOD автоматически.

Уровни детализации с Billboard

Формы могут содержать уровни детализации в виде billboard/imposter. Движок может использовать их для рендеринга удалённых объектов, если форма настроена соответствующим образом.

Это определяется данными формы и не настраивается непосредственно в TSStatic.

Отладочные поля

renderNormals 

"renderNormals": 1

Если значение больше нуля, отображает отладочные нормали для текущего уровня детализации.

Полезно для проверки:

  • Сломанных нормалей
  • Некорректного затенения
  • Зеркальной/неправильной геометрии
  • Проблем при экспорте

forceDetail

Принудительно задаёт уровень детализации; полезно для проверки LOD.

Рейкасты и выделение

TSStatic поддерживает рейкасты и генерацию списков полигонов.

В зависимости от контекста может использоваться:

  • Ограничивающий параллелепипед
  • Меши коллизий
  • Видимые меши
  • Текущий уровень детализации рендеринга
  • Детали линии видимости (LOS)

Это влияет на:

  • Выделение в редакторе
  • Физические запросы
  • Проецирование декалей
  • Рейкасты в геймплее
  • Коллизии

Перезагрузка ресурсов

TSStatic отслеживает изменения файла формы.

Если referenced файл формы изменяется, объект может пересоздать свою форму и обновить данные коллизий и материалов.

Это удобно при итеративной работе над артом уровня.

Полный пример 

{
  "class": "TSStatic",
  "name": "warehouse_01",
  "shapeName": "/levels/example/art/shapes/buildings/warehouse_01.dae",
  "position": [250, 120, 0],
  "rotationMatrix": [0, -1, 0, 1, 0, 0, 0, 0, 1],
  "scale": [1, 1, 1],
  "collisionType": "Collision Mesh",
  "decalType": "Collision Mesh",
  "dynamic": false,
  "playAmbient": true,
  "meshCulling": true,
  "originSort": false,
  "useInstanceRenderData": false,
  "isRenderEnabled": true,
  "canSaveDynamicFields": true
}

Когда использовать TSStatic, а когда Forest

Используйте TSStatic для индивидуально размещённых объектов:

  • Здания
  • Мосты
  • Барьеры
  • Крупный реквизит
  • Уникальные объекты
  • Объекты, требующие точного размещения
  • Объекты с индивидуальными настройками коллизий или материалов

Используйте систему Forest для большого количества повторяющихся объектов:

  • Деревья
  • Камни
  • Кусты
  • Пучки травы
  • Рассыпанный реквизит
  • Повторяющиеся ассеты

Элементы Forest, как правило, более легковесны при большом количестве объектов.

Лучшие практики

  • Используйте TSStatic для вручную размещённых статических мешей уровня.
  • Используйте элементы Forest для большого количества повторяющихся природных или рассыпанных ассетов.
  • По возможности используйте выделенные меши коллизий.
  • Избегайте коллизий по видимому мешу, если это не абсолютно необходимо.
  • Используйте collisionType: None для декоративных объектов без коллизий.
  • Размещайте начало координат объекта разумно, особенно для прозрачных объектов и сортировки.
  • Оставляйте dynamic отключённым, если это не требуется.
  • Позвольте движку выбирать LOD автоматически; избегайте forceDetail, кроме отладки.
  • Используйте данные рендеринга экземпляра только тогда, когда этого требует материал.
  • Сохраняйте стабильные имена целей материалов и сопоставляйте их через mapTo.
  • Используйте ассеты .dae/.cdae с правильными LOD и геометрией коллизий.

Распространённые проблемы

Объект не отображается

Возможные причины:

  • Неверный shapeName
  • Отсутствует файл формы
  • Форма не загружается
  • Объект скрыт
  • Настройки LOD/детализации скрывают его
  • Критические ошибки материалов делают объект невидимым

Объект отображается с предупреждающим материалом

Возможные причины:

  • Отсутствует определение материала
  • Несоответствие mapTo
  • Файл материала не загружен
  • Текстура отсутствует

Объект не имеет коллизий

Проверьте:

"collisionType": "Collision Mesh"

Также убедитесь, что форма действительно содержит данные меша или деталей коллизий.

Коллизия объекта слишком ресурсоёмка

Возможные причины:

  • Используется коллизия по видимому мешу
  • Меш коллизий слишком детализирован
  • Объект очень крупный или сложный
  • Слишком много объектов TSStatic с коллизиями

Декали проецируются некорректно

Проверьте:

"decalType": "Collision Mesh"

или используйте тип меша, лучше подходящий для отсечения декалей.

Фоновая анимация не воспроизводится

Убедитесь, что:

  • playAmbient имеет значение true
  • Форма содержит анимационную последовательность с именем ambient
  • Объект не скрыт

Цвет экземпляра не работает

Убедитесь, что:

  • useInstanceRenderData имеет значение true
  • Материал использует функции цвета экземпляра
  • Материал создан с умножением базового цвета/свечения/прозрачности на цвет экземпляра

Сортировка прозрачных объектов выглядит неправильно

Попробуйте originSort, а также проверьте:

  • Начало координат объекта
  • Режим смешивания материала
  • Разбивку геометрии
  • Порядок отрисовки

Итог

TSStatic — основной класс объектов для индивидуально размещённых статических 3D-мешей в уровнях.

Для большого количества повторяющихся природных или рассыпанных ассетов используйте систему Forest. Геометрия уровней BeamNG обычно должна размещаться либо как объекты TSStatic, либо как элементы Forest.

TSStatic напрямую ссылается на файл формы и поддерживает рендеринг, коллизии, декали, LOD, переназначение материалов, опциональную фоновую анимацию и данные рендеринга на уровне экземпляра.

См. также: Формат файла материалов , Prefab , Forest .
