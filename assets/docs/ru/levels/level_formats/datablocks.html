Этот документ описывает JSON-файлы даблоков BeamNG.

Даблоки — это повторно используемые определения данных движка. Они применяются во многих системах для описания общих ресурсов: частиц, эмиттеров, осадков, световых бликов, световых анимаций, звуков, фонового окружения, декалей и других повторно используемых объектов движка.

Даблок сам по себе обычно не является видимым объектом уровня. Вместо этого он представляет собой именованное определение, на которое могут ссылаться другие объекты, системы или игровой код.

Обзор

JSON-файл даблока — это JSON-объект, в котором каждый ключ верхнего уровня определяет один даблок.

Пример:

{
  "exampleData": {
    "name": "exampleData",
    "class": "SomeDatablockClass"
  }
}

Каждая запись даблока обычно содержит:

Поле Описание
name Имя объекта даблока. Обычно совпадает с ключом JSON.
class Класс/тип даблока.
class-specific fields Настройки, используемые данным типом даблока.

Ключ верхнего уровня и поле name должны в норме совпадать.

Именование файлов

Современные файлы даблоков обычно используют суффикс:

*.datablocks.json

Примеры:

art/datablocks/lights.datablocks.json
art/datablocks/sounds.datablocks.json
art/datablocks/audioProfiles.datablocks.json
art/datablocks/managedDatablocks.datablocks.json

Некоторые управляемые файлы даблоков используют специфичные для инструментов имена:

art/shapes/particles/managedParticleData.json
art/shapes/particles/managedParticleEmitterData.json

Специфичные для уровня даблоки могут также находиться внутри папок уровней:

levels/<levelName>/datablocks/custom.datablocks.json

Поведение при загрузке

Во время загрузки уровня основные даблоки загружаются до объектов уровня. Это позволяет объектам сцены и системам ссылаться на даблоки по имени.

Типичные базовые файлы включают:

art/shapes/particles/managedParticleData.json
art/shapes/particles/managedParticleEmitterData.json
art/datablocks/audioProfiles.datablocks.json
art/datablocks/sounds.datablocks.json
art/datablocks/player.datablocks.json
art/datablocks/datablockExec.datablocks.json
art/datablocks/lights.datablocks.json
art/datablocks/managedDatablocks.datablocks.json

При загрузке уровня движок также выполняет поиск в папке уровня файлов, соответствующих шаблонам вида:

*.datablocks.json
*Data.json

Они загружаются до создания объектов сцены уровня.

Устаревшие даблоки .cs

В устаревшем контенте даблоки могут быть определены в файлах TorqueScript .cs.

Примеры:

datablockExec.cs

Движок по-прежнему поддерживает некоторые пути загрузки устаревших даблоков .cs, однако новый контент должен использовать JSON-файлы даблоков.

JSON-файлы даблоков являются современной заменой многих устаревших определений даблоков TorqueScript.

Общая структура

Файл даблока содержит именованные записи:

{
  "datablockName": {
    "name": "datablockName",
    "class": "SomeDatablockClass",
    "fieldA": 1,
    "fieldB": "value"
  }
}

Минимальный пример:

{
  "TurretFireSound": {
    "name": "TurretFireSound",
    "class": "SFXProfile",
    "description": "AudioMaster",
    "filename": "art/sound/turret_firing.wav",
    "preload": true
  }
}

Распространённые классы даблоков

Распространённые классы даблоков включают:

Класс Назначение
ParticleData Определяет внешний вид и поведение отдельных частиц.
ParticleEmitterData Определяет поведение испускания частиц.
ParticleEmitterNodeData Определяет настройки узла эмиттера частиц.
PrecipitationData Определяет текстуры осадков.
LightFlareData Определяет элементы бликов линз/световых бликов.
LightAnimData Определяет поведение анимированной яркости, смещения или вращения источника света.
SFXProfile Определяет повторно используемый профиль звукового события/файла.
SFXAmbience Определяет данные фонового окружения или аудиоснимка.

Поля, специфичные для каждого класса, описаны отдельно.

Объекты сцены, которые обычно ссылаются на даблоки, включают Lights и Sound .

Ссылки между даблоками

Даблоки часто ссылаются друг на друга по имени.

Пример:

{
  "exampleEmitter": {
    "name": "exampleEmitter",
    "class": "ParticleEmitterData",
    "particles": "exampleParticle"
  }
}

Здесь exampleEmitter ссылается на даблок ParticleData с именем exampleParticle.

Если указанный даблок отсутствует, эффект может не работать или использовать резервное поведение.

Пути к текстурам и аудио

Многие даблоки ссылаются на текстуры или аудиоресурсы.

Примеры:

"textureName": "art/shapes/particles/steam.png"
"flareTexture": "core/art/special/lensFlareSheet0.color.png"
"dropTexture": "art/shapes/particles/Particle_rain_drop.dds"
"filename": "art/sound/turret_firing.wav"
"fileName": "event:>UI>Missions>Countdown_Go"

Используйте корректные пути и убедитесь, что указанные файлы включены в мод или уровень.

Некоторые классы даблоков используют fileName, другие могут использовать filename. Используйте существующий контент игры или документацию по конкретному классу в качестве ориентира.

Постоянные идентификаторы

Некоторые сохранённые даблоки содержат:

"persistentId": "c98465df-dd27-48d0-b855-3f5267b18400"

Постоянные идентификаторы — это идентификаторы редактора/движка, используемые для обеспечения постоянства объектов.

При создании даблоков вручную избегайте копирования одного и того же persistentId в другой, не связанный с ним даблок.

Не дублируйте значения persistentId между даблоками. При создании новых даблоков вручную опустите это поле и позвольте редактору сгенерировать его.

Значения null и заполнители

Сохранённые файлы даблоков могут содержать значения null или пустые записи-заполнители.

Пример:

{
  "elementDist": null,
  "elementRect": null,
  "elementScale": null
}

или:

{}

Обычно они представляют неиспользуемые слоты массива или значения по умолчанию, записанные редактором. Инструменты, как правило, могут игнорировать неиспользуемые записи-заполнители, если только конкретный класс даблока не требует их наличия.

Рабочий процесс в редакторе

Даблоки можно создавать и управлять ими с помощью инструментов редактора.

При создании даблока:

  1. Редактор создаёт объект выбранного класса даблока.
  2. Опционально копирует поля из другого даблока.
  3. Регистрирует объект.
  4. Добавляет его в набор даблоков.
  5. Помечает его как изменённый в менеджере постоянства.
  6. Сохраняет его в JSON-файл даблока.

Редактор отслеживает изменённые даблоки и может сохранять только модифицированные объекты.

Файл управляемого даблока по умолчанию

Если пользовательское имя файла не задано, даблоки, созданные редактором, могут быть сохранены в управляемый файл вида:

art/datablocks/managedDatablocks.json

или в управляемый файл, специфичный для класса/инструмента.

Для упорядоченного контента предпочтительно сохранять связанные даблоки в файлы с понятными именами.

Пример:

levels/example/datablocks/traffic_lights.datablocks.json
levels/example/datablocks/weather_effects.datablocks.json
levels/example/datablocks/custom_sounds.datablocks.json

Минимальный пример файла 

{
  "exampleSound": {
    "name": "exampleSound",
    "class": "SFXProfile",
    "description": "AudioDefault3D",
    "fileName": "art/sound/example.ogg"
  },

  "exampleFlare": {
    "name": "exampleFlare",
    "class": "LightFlareData",
    "flareTexture": "core/art/special/lensFlareSheet0.color.png",
    "overallScale": 1
  }
}

Рекомендации

  • Используйте JSON-файлы даблоков для нового контента.
  • Поддерживайте уникальность имён даблоков.
  • Следите за согласованностью ключа верхнего уровня и поля name.
  • Используйте описательные имена, отражающие назначение или систему.
  • Загружайте/создавайте referenced даблоки до их потребителей.
  • Избегайте копирования значений persistentId между несвязанными даблоками.
  • Используйте пути к ресурсам, работающие как в обычной игре, так и в запакованных модах.
  • По возможности предпочитайте общие/совместно используемые даблоки.
  • Используйте инструменты редактора для создания и сохранения даблоков, когда они доступны.
  • Удаляйте неиспользуемые даблоки, чтобы снизить нагроможденность и время загрузки.
  • Группируйте связанные даблоки в файлы с понятными именами.

Распространённые проблемы

Даблок не найден

Возможные причины:

  • Файл не был загружен.
  • Неверный суффикс файла.
  • Неверное значение name.
  • Опечатка в имени referenced даблока.
  • Ссылка на даблок сделана до его загрузки.

Referenced ресурс отсутствует

Возможные причины:

  • Неверный путь к текстуре/аудио.
  • Файл не включён в мод или уровень.
  • Путь работает локально, но не из запакованного мода.
  • Неверный регистр символов в системах с чувствительностью к регистру.

Даблок загружается, но не имеет эффекта

Возможные причины:

  • Ни один объект или система не ссылается на него.
  • Отсутствуют обязательные поля, специфичные для класса.
  • Дочерний referenced даблок отсутствует.
  • Даблок был сохранён в неверный файл.

Дублирующиеся имена даблоков

Каждое имя даблока должен использовать только один объект. Переименуйте один из них и обновите ссылки.

Конфликт постоянного идентификатора

Удалите продублированные поля persistentId и позвольте редактору заново их сгенерировать.

Резюме

JSON-файлы даблоков хранят повторно используемые определения данных движка.

Они могут определять частицы, эмиттеры, осадки, световые блики, световые анимации, звуки, фоновое окружение, декали и другие общие ресурсы.

Современные даблоки должны храниться в JSON-файлах вида:

*.datablocks.json

или в управляемых JSON-файлах, специфичных для инструментов.

Объекты уровня и игровые системы ссылаются на даблоки по имени.
