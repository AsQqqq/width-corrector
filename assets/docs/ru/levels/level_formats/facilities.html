На этой странице описаны файлы объектов уровня (facilities).

Объекты (facilities) определяют именованные игровые локации: гаражи, заправочные станции, автосалоны, компьютеры, частных продавцов, службы доставки и трассы для дрэг-рейсинга. Они придают игровой смысл местам на уровне; пространственные данные при этом обычно берутся из файлов .sites.json и объектов сцены.

Информацию об установке заправочных и зарядных станций см. в разделе Fuel Stations .

Расположение файлов

Данные об объектах могут читаться из файла info.json уровня, а также из файлов объектов в папке facilities уровня:

levels/<levelName>/info.json
levels/<levelName>/facilities/*.facilities.json
levels/<levelName>/facilities/**/*.facilities.json

Примеры:

levels/west_coast_usa/facilities/facilities.facilities.json
levels/west_coast_usa/facilities/dealerships.facilities.json
levels/west_coast_usa/facilities/delivery/shops.facilities.json

Некоторые папки функций также могут содержать файлы, подобные файлам объектов, для своих собственных систем — например, файлы объектов трасс для дрэг-рейсинга. Воспринимайте их как данные, специфичные для конкретной функции, если только они не используются в системе объектов уровня.

Поведение при загрузке

Система объектов формирует список объектов для текущего уровня, считывая:

  1. Файл info.json уровня.
  2. Все файлы *.facilities.json в папке levels/<levelName>/facilities/, включая вложенные подпапки.

Это означает, что уровень может хранить простые записи объектов в info.json, а более крупные наборы функций выносить в отдельные файлы:

facilities.facilities.json
dealerships.facilities.json
delivery/shops.facilities.json
delivery/warehouses.facilities.json

Разделение файлов удобно, когда разные системы или разработчики отвечают за разные типы объектов. Идентификаторы объектов по-прежнему должны быть стабильными и уникальными в пределах своего типа после загрузки всех файлов.

Создание файлов объектов

Универсального редактора для всех типов объектов не существует. Рекомендуемый рабочий процесс: создайте пространственные данные в World Editor, затем напишите или отредактируйте JSON объекта, который ссылается на эти пространственные данные.

Для большинства типов объектов:

  1. Создайте или выберите объекты сцены, используемые объектом, — например, игровые зоны, точки интереса (POI), двери, насосы или видимые вывески.
  2. Создайте необходимые зоны и парковочные места в файле .sites.json — обычно это facilities.sites.json или соседний файл сайтов конкретной функции.
  3. Создайте или отредактируйте файл *.facilities.json в папке levels/<levelName>/facilities/.
  4. Добавьте запись объекта в нужный массив верхнего уровня: garages, dealerships или deliveryProviders.
  5. Укажите ссылки на файл сайтов, имена зон, имена парковочных мест и имена объектов сцены.
  6. Перезагрузите уровень или перезагрузите игровую систему, использующую данный объект.
  7. Протестируйте объект в режиме свободной езды, карьере, на большой карте или в пользовательском интерфейсе конкретной функции.

Используйте специализированный редактор или страницу функции там, где они есть. Например, заправочные станции имеют выделенный процесс настройки, а трассы для дрэг-рейсинга рекомендуется поддерживать через инструменты дрэг-рейсинга там, где это возможно.

Написание файлов объектов вручную

При написании вручную начинайте с минимально допустимой записи и добавляйте поля, специфичные для функции, только после того, как объект появится в игре.

Рекомендуемый порядок:

  1. Добавьте id, name, description и preview.
  2. Добавьте sitesFile, zoneNames и parkingSpotNames.
  3. Убедитесь, что объект обнаруживается и имеет пригодную позицию.
  4. Добавьте поля, специфичные для типа: фильтры автосалона, двери, метаданные доставки или данные насосов заправочной станции.
  5. Протестируйте функцию, использующую эти поля.

Избегайте копирования крупной готовой записи объекта с изменением только идентификатора. Готовые записи могут содержать поля карьеры, экономики, доставки или покупки транспортных средств, которые не нужны для более простого объекта.

Именование и организация

Используйте стабильные идентификаторы и описательные имена:

garage_downtown
dealership_city_used
delivery_warehouse_west
computer_service_station

Для больших уровней разбивайте файлы по функции или ответственному разработчику:

facilities/facilities.facilities.json
facilities/dealerships.facilities.json
facilities/delivery/warehouses.facilities.json
facilities/delivery/shops.facilities.json

По возможности храните связанные изображения предварительного просмотра и файлы сайтов конкретной функции рядом с файлом объекта.

Типы объектов

Общая система объектов распознаёт следующие массивы верхнего уровня:

Тип объекта Имя массива
Гараж garages
Заправочная / зарядная станция gasStations
Автосалон dealerships
Компьютер computers
Частный продавец privateSellers
Служба доставки deliveryProviders
Трасса для дрэг-рейсинга dragstrips

Один файл может содержать один или несколько таких массивов.

Общая структура

Большинство записей объектов имеют небольшой общий основной блок:

{
  "garages": [
    {
      "id": "downtownGarage",
      "name": "levels.example.facilities.downtownGarage",
      "description": "levels.example.facilities.downtownGarage.description",
      "preview": "downtownGarage.jpg",
      "sitesFile": "facilities.sites.json",
      "zoneNames": ["downtownGarageZone"],
      "parkingSpotNames": ["downtownGarageParking01"]
    }
  ]
}

Поля, специфичные для функции, допускаются и зависят от типа объекта. Например, автосалоны могут иметь фильтры транспортных средств и данные тест-драйва, тогда как заправочные станции определяют поддерживаемые типы топлива и пары объектов насосов.

Заправочная станция

Заправочные станции используют то же семейство файлов объектов, однако настройка насосов и типов топлива имеет свою специфику. Подробную структуру см. в разделе Fuel Stations .

Общие поля объектов

Многие записи объектов используют одинаковые паттерны полей:

Поле Назначение
id Стабильный идентификатор объекта, используемый геймплеем, точками интереса, сохранениями и скриптами.
name Отображаемое имя или ключ перевода.
description Отображаемое описание или ключ перевода.
preview Изображение предварительного просмотра, отображаемое в интерфейсе там, где поддерживается.
sitesFile Файл сайтов, используемый для разрешения зон и парковочных мест.
zoneNames Зоны, связанные с объектом.
parkingSpotNames Парковочные места, связанные с объектом.
icon / playModeIconName Необязательные идентификаторы иконок, используемые интерфейсом или маркерами карты.
tags Необязательные метаданные для фильтрации или поведения, специфичного для функции.

Загрузчик объектов заполняет некоторые значения по умолчанию. Если поле id отсутствует, идентификатор может быть сгенерирован на основе пути к файлу и индекса, однако это не рекомендуется для авторского контента, поскольку такой идентификатор нестабилен при перемещении файлов.

Пространственные ссылки

Объекты обычно ссылаются на пространственные данные, а не хранят все позиции непосредственно.

Ссылка Обычно разрешается в
sitesFile Файл .sites.json, обычно facilities.sites.json или файл рядом с файлом объекта.
zoneNames Зоны в указанном файле сайтов.
parkingSpotNames Парковочные места в указанном файле сайтов.
doors Пары объектов сцены, используемые автосалонами, компьютерами или другими точками взаимодействия.
pumps Пары объектов игровой зоны и точки интереса заправочной станции.

Если sitesFile отсутствует, загрузчик по возможности использует facilities.sites.json.

sitesFile может быть одним файлом или, в расширенных конфигурациях, списком файлов. Используйте это, когда объект должен разрешать парковочные места или зоны, распределённые по нескольким файлам сайтов. Применяйте такой подход редко: единственный соседний файл сайтов проще валидировать.

Примечания по типам объектов

Гаражи

Гаражи обычно используют zoneNames и parkingSpotNames. Они задействованы в режиме свободной езды, инвентаре карьеры, процессах восстановления/телепортации и логике ближайших сервисов. Убедитесь, что парковочные места достаточно велики и правильно ориентированы для предполагаемых транспортных средств.

Для создания простого гаража создайте как минимум одну зону и одно парковочное место, затем добавьте запись гаража, ссылающуюся на них. Протестируйте телепортацию/восстановление с несколькими типоразмерами транспортных средств.

Автосалоны и частные продавцы

Автосалоны и частные продавцы нередко включают фильтры транспортных средств, данные о сборах, настройки тест-драйва и ссылки на двери или точки интереса. Поля фильтров транспортных средств используются логикой покупки автомобилей в карьере, поэтому при редактировании инструментами сохраняйте неизвестные ключи фильтров.

Для создания автосалона сначала создайте объекты дверей/зон взаимодействия и парковочные места, затем добавьте запись автосалона. Добавляйте фильтры только после того, как автосалон корректно появится и откроется.

Компьютеры

Объекты-компьютеры — это точки взаимодействия для систем карьеры: покупок, тюнинга, страхования и других меню. Они обычно используют ссылки на двери/зоны, а не данные о парковочных местах.

Для компьютеров важнее всего, чтобы указанный объект взаимодействия существовал и был доступен игроку.

Службы доставки

Службы доставки используются логикой генерации и организации доставок. Они обычно объединяют метаданные объекта с сайтами и парковочными местами, специфичными для доставки. Вложенные файлы в папке facilities/delivery/ распространены в готовом контенте.

Обслуживать службы доставки проще всего, когда их файл сайтов находится рядом с файлом провайдера. Это позволяет держать зоны доставки, места получения/выдачи и записи объектов вместе.

Трассы для дрэг-рейсинга

Объекты трасс для дрэг-рейсинга связывают интерфейс свободной езды/карьеры с данными дрэг-рейсинга. Игровой процесс дрэг-рейсинга по-прежнему использует собственные файлы в папке dragstrips/; запись объекта является уровнем интеграции с картой/карьерой.

Заправочные станции

Для заправочных станций и точек зарядки электромобилей предусмотрена отдельная страница настройки. Информацию о парах насосов, типах топлива и деталях цен/отображения см. в разделе Fuel Stations .

Пути к изображениям предварительного просмотра

Изображения предварительного просмотра обычно хранятся рядом с файлом объекта или в папке уровня:

levels/<levelName>/facilities/<previewImage>.jpg
levels/<levelName>/<previewImage>.jpg

Если изображение предварительного просмотра отсутствует, интерфейс может использовать универсальную заглушку.

Связь с сайтами и точками интереса

Объекты часто отображаются в режиме свободной езды, карьере и на большой карте. Отображаемый маркер или взаимодействие может поступать из:

  • Записи объекта.
  • Зон или парковочных мест в файле .sites.json.
  • Объектов BeamNGGameplayArea и BeamNGPointOfInterest.
  • Систем, специфичных для функции: заправочных станций, служб доставки, автосалонов или трасс для дрэг-рейсинга.

Используйте Sites для данных сайтов и Gameplay Areas and Points of Interest для вспомогательных объектов сцены.

Устранение неполадок

Объект не отображается

Проверьте:

  • Файл находится в info.json или в папке levels/<levelName>/facilities/.
  • Расширение файла — *.facilities.json.
  • Объект находится в правильном массиве верхнего уровня, например garages или dealerships.
  • Идентификатор объекта уникален в пределах своего типа.
  • Обязательные поля, специфичные для функции, присутствуют.

Объект отображается, но не имеет пригодной позиции

Проверьте:

  • sitesFile разрешается в ожидаемый файл .sites.json.
  • Каждый элемент zoneNames существует в файле сайтов.
  • Каждый элемент parkingSpotNames существует в файле сайтов.
  • Ссылки на объекты сцены в полях doors или pumps существуют.

У объекта отсутствует изображение предварительного просмотра

Пути к изображениям предварительного просмотра разрешаются относительно файла объекта и папки уровня. По возможности храните изображения предварительного просмотра рядом с файлом объекта.

Контрольный список валидации

  • Идентификаторы объектов уникальны в пределах своего типа.
  • Имена и описания, отображаемые пользователю, используют ключи перевода там, где это уместно.
  • Изображения предварительного просмотра существуют или имеют приемлемые заглушки.
  • Указанные пути sitesFile существуют.
  • Указанные зоны и парковочные места существуют в загруженном файле сайтов.
  • Указанные объекты сцены, двери, насосы или точки интереса существуют в сцене уровня.
  • Поля, специфичные для функции, протестированы в системе, которая их использует.
  • Файлы объектов остаются валидными после перемещения записей между файлами или подпапками.
