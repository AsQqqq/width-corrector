Этот документ описывает файлы BeamNG .sites.json.

Файлы sites хранят пространственные метаданные, связанные с игровым процессом, такие как:

  • Локации
  • Зоны
  • Места парковки
  • Теги
  • Пользовательские поля

Они используются системами геймплея, миссиями, трафиком, логикой парковки, объектами, точками интереса и инструментами редактора для идентификации именованных точек и областей уровня.

Обзор

Файл .sites.json определяет именованные пространственные данные. В отличие от обычных объектов сцены, sites не отрисовываются как геометрия уровня. Вместо этого они описывают полезные игровые позиции, области и места парковки.

Основные варианты использования:

  • Точки старта и финиша миссий
  • Именованные городские районы или регионы карты
  • Зоны объектов, такие как гаражи или автосалоны
  • Размещение мест парковки
  • Зоны появления транспортных средств
  • Зоны доставки/разгрузки
  • Фильтрация игровых зон
  • Метаданные для трафика, парковки и процедурных миссий
  • Границы миссий

Sites обычно редактируются с помощью:

World Editor → Gameplay → Sites Editor
Sites — это метаданные геймплея, а не геометрия сцены. Они, как правило, потребляются скриптами и системами геймплея, а не отрисовываются напрямую.

Расположение файлов

Файлы sites используют суффикс:

*.sites.json

Распространённые примеры:

levels/<levelName>/city.sites.json
levels/<levelName>/facilities.sites.json
levels/<levelName>/garages.sites.json
gameplay/missions/<levelName>/<missionType>/<missionName>/bounds.sites.json
/gameplay/*.sites.json

Менеджер sites может сканировать файлы .sites.json внутри папок уровней и по пути:

/gameplay/

Это позволяет использовать как специфичные для уровня, так и общие игровые sites.

Распространённые файлы sites

Файл Назначение
city.sites.json Файл sites уровня по умолчанию. Обычно используется для общих зон, локаций и мест парковки.
facilities.sites.json Sites, на которые ссылаются объекты: гаражи, заправочные станции, автосалоны, компьютеры, дрэг-стрипы и т. д.
garages.sites.json Зоны и места парковки, связанные с гаражами/карьерой; часто загружаются системами, которым нужны списки гаражных зон.
bounds.sites.json Данные о границах миссий; часто используются старыми/конвертированными форматами миссий.
/gameplay/*.sites.json Общие/глобальные файлы sites, используемые системами геймплея или инструментами.

Файл sites уровня по умолчанию

При запуске уровня системы геймплея обычно пытаются загрузить файл sites по умолчанию для текущей карты:

<level folder>/city.sites.json

Например:

levels/west_coast_usa/city.sites.json

Если файл существует, он загружается и кэшируется менеджером sites.

Этот файл по умолчанию обычно используется следующими системами:

  • Вспомогательные средства для метаданных города/уровня
  • Парковка
  • Случайный выбор места парковки
  • Поиск зоны по позиции
  • Метаданные групп транспортных средств
  • Утилиты миссий и геймплея

Примеры других файлов sites

facilities.sites.json

Объекты обычно определяются в файлах info.json или *.facilities.json, однако эти определения объектов нередко ссылаются на зоны или места парковки, хранящиеся в файле .sites.json.

Например, объект гаража может ссылаться на:

"sitesFile": "facilities.sites.json",
"zoneNames": ["garage_downtown"],
"parkingSpotNames": ["garage_downtown_01", "garage_downtown_02"]

Файл sites содержит фактические данные о зонах и местах парковки.

Файл объектов (facilities) определяет метаданные объекта. Файл .sites.json определяет пространственные зоны и места парковки, используемые этим объектом.

garages.sites.json

Некоторые системы геймплея/карьеры загружают файл sites из настроенной папки, обычно:

garages.sites.json

Это используется для формирования списков зон, связанных с гаражами. Например, инструменты могут загружать все названия зон из этого файла, чтобы дизайнеры могли выбирать допустимые начальную и конечную гаражные зоны.

bounds.sites.json

Некоторые данные миссий используют файл bounds.sites.json для хранения границ миссий.

Рабочий процесс портирования миссии может конвертировать:

bounds.sites.json

в более новый формат границ.

В этом случае файл sites обычно содержит одну или несколько зон, причём первая зона может использоваться в качестве границы миссии.

Основные типы данных

Файл sites может содержать три основных типа пространственных элементов:

Тип Описание
Локация (Location) Именованная точка или небольшая область. Полезна для ориентиров, точек миссий и ссылок.
Зона (Zone) Именованный полигон/область, используемый для проверки того, находится ли позиция внутри региона.
Место парковки (Parking Spot) Позиция и ориентация, в которых транспортные средства могут быть размещены, заспавнены или проверены на наличие парковки.

Каждый тип также может иметь:

  • Имя
  • Порядок сортировки
  • Теги
  • Пользовательские поля

Локации

Локации — это именованные точки или небольшие пространственные маркеры.

Они полезны для:

  • Точек старта миссий
  • Точек финиша миссий
  • Ориентиров
  • Точек доставки
  • Ссылок для ИИ/навигации
  • Опорных точек камеры или геймплея

Локация обычно хранит:

Поле Описание
name Название локации.
pos Мировая позиция.
radius Необязательный размер/область влияния.
tags Необязательный список тегов.
customFields Необязательные метаданные геймплея.
sortOrder Значение порядка в редакторе.

Примеры названий локаций:

gas_station
police_station
industrial_delivery_01
race_start

Зоны

Зоны определяют именованные области карты.

Они обычно используются для:

  • Районов
  • Границ миссий
  • Зон объектов
  • Зон гаражей/автосалонов
  • Зон доставки
  • Фильтрации групп транспортных средств
  • Группировки мест парковки
  • Логики регионов появления
  • Проверки игровых областей

Игровой код может запрашивать зоны, содержащие позицию, например:

getZonesForPosition(pos)

Зоны особенно полезны, когда системе нужно знать «где» что-то находится, а не только ближайшую точку.

Границы зон

Зоны — это полигональные области, образованные вершинами. Данные некоторых зон могут также включать вертикальные ограничения, такие как верхняя и нижняя плоскости.

Концептуальная зона может содержать:

{
  "name": "downtown",
  "vertices": [
    {"pos": [0, 0, 0]},
    {"pos": [100, 0, 0]},
    {"pos": [100, 100, 0]},
    {"pos": [0, 100, 0]}
  ],
  "top": {
    "active": false,
    "pos": [0, 0, 10],
    "normal": [0, 0, 1]
  },
  "bot": {
    "active": false,
    "pos": [0, 0, -10],
    "normal": [0, 0, -1]
  }
}
Точная сериализованная структура генерируется редактором Sites и системой игровых sites. По возможности используйте редактор для создания и сохранения файлов.

Приоритет зон

Зоны могут использовать пользовательское поле с именем:

priority

Зоны с более высоким приоритетом предпочтительнее, когда несколько зон содержат одну и ту же позицию.

Пример:

"customFields": {
  "priority": 10
}

Это полезно, когда меньшая, более специфичная область находится внутри более крупной общей зоны.

Пример:

City
 └── Downtown
      └── Dealership

Если все три зоны содержат одну и ту же позицию, системы могут выбирать зону с наивысшим приоритетом.

Пользовательские поля зон

Зоны могут хранить метаданные геймплея в пользовательских полях.

Часто используемое поле:

vehicleGroup

Система также поддерживает суффиксы для нескольких записей:

vehicleGroup
vehicleGroup1
vehicleGroup2
vehicleGroup3
vehicleGroup4

Это позволяет одной зоне ссылаться на несколько групп транспортных средств или записей метаданных.

Пример:

"customFields": {
  "priority": 5,
  "vehicleGroup": "city",
  "vehicleGroup1": "traffic",
  "vehicleGroup2": "parked"
}

Игровой код может объединять эти поля из зон с наивысшим приоритетом, соответствующих условию.

Места парковки

Места парковки определяют позиции размещения транспортных средств.

Они используются для:

  • Припаркованного трафика
  • Спауна транспортных средств в миссиях
  • Быстрого перемещения в гараж
  • Пунктов назначения доставки
  • Позиций начала/конца сценариев
  • Случайного выбора места парковки
  • Валидации парковки

Место парковки обычно хранит:

Поле Описание
name Название места парковки.
pos Мировая позиция.
rot Ориентация транспортного средства.
scl Размер/габариты места парковки.
tags Необязательный список тегов.
customFields Необязательные метаданные геймплея.
sortOrder Значение порядка в редакторе.

Места парковки могут быть связаны с зонами. Например, место парковки внутри зоны может наследовать метаданные района, объекта или геймплея из этой зоны.

Теги мест парковки

Места парковки могут использовать теги для управления поведением парковки.

Распространённые теги:

Тег Описание
banned Система парковки не должна использовать это место.
forwards Предпочтительная парковка передом.
backwards Предпочтительная парковка задом.
perfect Избегать случайного смещения/поворота при парковке.
street Обозначает место парковки у обочины. Может сохранять отслеживание, чтобы ИИ мог объезжать транспортное средство.
nightTime Место чаще используется ночью.
dayTime Место чаще используется днём.

Они в основном используются системами динамической парковки и трафика.

Пользовательские поля мест парковки

Места парковки могут использовать пользовательские поля, такие как:

probability

Пример:

"customFields": {
  "probability": 0.5
}

Это управляет тем, насколько вероятно использование места системами, расставляющими припаркованные транспортные средства.

Система парковки также может корректировать вероятность в зависимости от времени суток с помощью тегов nightTime и dayTime.

Случайный выбор места парковки

Утилиты геймплея могут выбирать случайные пары мест парковки.

Например, одна функция может возвращать:

start parking spot
destination parking spot
start zone
destination zone

с минимальным расстоянием между стартом и пунктом назначения.

Это полезно для:

  • Процедурных миссий
  • Маршрутов доставки
  • Случайных заданий по вождению
  • Карьерного геймплея

Теги

Sites поддерживают теги для:

  • Локаций
  • Зон
  • Мест парковки

Теги используются для фильтрации и организации данных.

Примеры тегов:

delivery
garage
fuel
parking
industrial
urban
offroad
race
banned
street
nightTime
dayTime

В редакторе Sites есть отдельные вкладки для:

  • Тегов локаций (Location Tags)
  • Тегов зон (Zone Tags)
  • Тегов мест парковки (Parking Spot Tags)

Пользовательские поля

Пользовательские поля позволяют системам геймплея прикреплять произвольные метаданные к sites.

Примеры:

"customFields": {
  "priority": 10,
  "vehicleGroup": "traffic_city",
  "missionType": "delivery",
  "difficulty": "easy",
  "probability": 0.75
}

Пользовательские поля обычно используются системами, которым нужны специфичные для карты данные без их жёсткого кодирования в скриптах.

Объекты и sites

Объекты используют файлы sites для получения пространственных данных.

Определения объектов могут ссылаться на:

"sitesFile": "facilities.sites.json",
"zoneNames": ["dealership_downtown"],
"parkingSpotNames": ["dealership_downtown_01"]

Затем система объектов загружает эти файлы sites и выполняет поиск:

  • Зон по имени
  • Мест парковки по имени

Это используется для:

  • Гаражей
  • Заправочных станций
  • Автосалонов
  • Компьютеров
  • Частных продавцов
  • Поставщиков услуг доставки
  • Дрэг-стрипов

Границы миссий

Некоторые старые форматы миссий хранят границы в:

bounds.sites.json

Эти файлы обычно содержат зоны, представляющие игровую/миссионную область.

Инструменты конвертации миссий могут преобразовывать эти данные в более новые форматы границ/путей/маршрутов.

Типичная конвертация:

bounds.sites.json → boundary file
race.race.json → path file
race.race.json startPositions → starting position file

Редактор Sites

Редактор Sites доступен в Мировом редакторе (World Editor):

World Editor → Gameplay → Sites Editor

Редактор предоставляет вкладки для:

  • Локаций (Locations)
  • Зон (Zones)
  • Мест парковки (Parking Spots)
  • Тегов локаций (Location Tags)
  • Тегов зон (Zone Tags)
  • Тегов мест парковки (Parking Spot Tags)

Редактор позволяет:

  • Создавать новые файлы sites
  • Загружать существующие файлы .sites.json
  • Сохранять файлы sites
  • Сортировать локации по имени
  • Сортировать зоны по имени
  • Сортировать места парковки по имени
  • Перечислять дублирующиеся имена мест парковки
  • Переименовывать места парковки на основе содержащей их зоны

Сохранение

При сохранении из редактора Sites путь к файлу сохраняется в объекте sites:

sites.dir
sites.name

Файл записывается в формате JSON с использованием:

jsonWriteFile(path, data, true)

По возможности используйте редактор для сохранения, так как он гарантирует, что сериализованная структура соответствует ожиданиям системы игровых sites.

Загрузка и кэширование

Менеджер sites загружает файлы sites через:

gameplay_sites_sitesManager.loadSites(filepath)

Загруженные файлы кэшируются по пути. Если тот же файл запрашивается повторно, используется кэшированный объект sites, если только не запрошена принудительная перезагрузка.

Кэш очищается по окончании миссии.

Обнаружение менеджером

Менеджер sites может сканировать все уровни в поисках файлов .sites.json.

Он хранит список файлов sites для каждого уровня:

sitesByLevel[levelName] = { ... }

Редактор Sites может отображать их через:

File → Sites in Manager...

Это упрощает поиск и загрузку файлов sites с других уровней или из /gameplay.

Упрощённая структура

Точная сериализованная структура генерируется системой игровых sites, однако концептуально файл sites содержит:

{
  "name": "city",
  "locations": [],
  "zones": [],
  "parkingSpots": [],
  "tags": {},
  "customFields": {}
}

Упрощённый концептуальный пример:

{
  "name": "city",
  "locations": [
    {
      "name": "gas_station",
      "pos": [100, 200, 0],
      "radius": 10,
      "tags": ["fuel", "service"]
    }
  ],
  "zones": [
    {
      "name": "downtown",
      "vertices": [
        {"pos": [0, 0, 0]},
        {"pos": [100, 0, 0]},
        {"pos": [100, 100, 0]},
        {"pos": [0, 100, 0]}
      ],
      "customFields": {
        "priority": 5,
        "vehicleGroup": "city"
      }
    }
  ],
  "parkingSpots": [
    {
      "name": "downtown_parking_01",
      "pos": [50, 50, 0],
      "rot": [0, 0, 0, 1],
      "scl": [2.5, 5.5, 1],
      "tags": ["parking", "street"],
      "customFields": {
        "probability": 0.75
      }
    }
  ]
}
Этот пример демонстрирует концептуальную модель данных. Для точного форматирования создайте и сохраните файл с помощью редактора Sites.

Рекомендуемый рабочий процесс

  1. Откройте Мировой редактор (World Editor).
  2. Откройте Gameplay → Sites Editor.
  3. Создайте или загрузите файл .sites.json.
  4. Добавьте локации, зоны и места парковки.
  5. При необходимости добавьте теги и пользовательские поля.
  6. Используйте инструменты сортировки/перечисления для поддержания порядка данных.
  7. Сохраните в соответствующий файл:
    • city.sites.json — для sites уровня по умолчанию
    • facilities.sites.json — для пространственных данных объектов
    • garages.sites.json — для данных зон гаражей/карьеры
    • bounds.sites.json — для данных о границах миссий, при необходимости

Рекомендации по именованию

Используйте чёткие и стабильные имена.

Хорошие примеры:

downtown
industrial_zone
gas_station_01
garage_downtown
parking_downtown_001
delivery_port_03
race_start_city

Избегайте:

test
new
zone1
asdf
parking

Для мест парковки часто полезны пронумерованные имена:

parking_downtown_001
parking_downtown_002
parking_downtown_003

Редактор включает инструмент Enumerate Parking Spots для работы с дублирующимися именами.

Рекомендации

  • Используйте city.sites.json в качестве файла sites уровня по умолчанию.
  • При необходимости используйте отдельные файлы sites для специализированных данных, например facilities.sites.json или garages.sites.json.
  • Используйте зоны для крупных именованных областей.
  • Используйте локации для конкретных точек интереса.
  • Используйте места парковки для размещения транспортных средств.
  • Используйте теги для фильтрации.
  • Используйте пользовательские поля для метаданных геймплея.
  • Используйте priority для перекрывающихся зон.
  • Сохраняйте имена стабильными, как только от них зависят миссии, объекты или данные карьеры.
  • По возможности используйте редактор Sites вместо ручного редактирования.
  • Сохраняйте и тестируйте после крупных изменений.
  • Не удаляйте и не переименовывайте зоны/места парковки, на которые ссылаются другие файлы.

Распространённые проблемы

Файл sites не загружается

Возможные причины:

  • Файл не существует
  • Недопустимый JSON
  • Неверное расширение файла
  • Файл не назван по шаблону *.sites.json
  • Файл находится не по ожидаемому пути уровня/геймплея

Sites по умолчанию не загружаются

Убедитесь, что уровень содержит:

city.sites.json

в папке уровня.

Зона объекта или место парковки отсутствует

Возможные причины:

  • Объект ссылается на неверный sitesFile
  • zoneNames не совпадают с именами зон в файле sites
  • parkingSpotNames не совпадают с именами мест парковки в файле sites
  • Путь к файлу sites неверен

Место парковки не найдено

Возможные причины:

  • Имя места парковки изменилось
  • Файл не загружен
  • Место парковки удалено
  • Теги или пользовательские поля не соответствуют ожидаемым фильтрам
  • Место парковки помечено тегом banned

Припаркованные автомобили не появляются в ожидаемых местах

Проверьте:

banned
probability
dayTime
nightTime
street
perfect
forwards
backwards

Также убедитесь, что транспортное средство помещается в габариты места парковки.

Приоритет зоны работает не так, как ожидается

Проверьте пользовательское поле зоны:

priority

Оно должно быть числовым.

Данные группы транспортных средств отсутствуют

Проверьте пользовательские поля зоны:

vehicleGroup
vehicleGroup1
vehicleGroup2
vehicleGroup3
vehicleGroup4

Места парковки имеют дублирующиеся имена

Используйте:

Tools → Enumerate Parking Spots

в редакторе Sites.

Итоги

Файлы .sites.json хранят метаданные игровой карты.

Они определяют:

  • Локации
  • Зоны
  • Места парковки
  • Теги
  • Пользовательские поля

Файл sites уровня по умолчанию обычно:

city.sites.json

Другие распространённые примеры включают:

facilities.sites.json
garages.sites.json
bounds.sites.json

Sites используются системами геймплея для поиска именованных мест, определения зон по позиции, выбора мест парковки, поддержки объектов и применения специфичных для карты метаданных, таких как группы транспортных средств и приоритеты.
